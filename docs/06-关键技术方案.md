## 一、出价并发(系统的"心脏")⭐

### 1.1 问题
100 个用户同时出价,要保证:最高价单调递增、不丢、不重、P99 < 500ms。

### 1.2 方案演进

| 方案 | 思路 | 问题 |
|------|------|------|
| A 朴素 | 读-改-写 | 必然超出价,必坏 |
| B 悲观锁 | `SELECT FOR UPDATE` | 串行,QPS 低 |
| C 乐观锁 | `version` CAS | 高并发失败率高 |
| **D Redis+Lua** ⭐ | Redis 端原子比较+更新+入队 | 本项目首选 |

### 1.3 Lua 脚本(`bid.lua`)

```lua
-- KEYS[1] = auction:price:<itemId>
-- KEYS[2] = auction:bid:queue:<itemId>
-- KEYS[3] = auction:idem:<requestId>
-- ARGV[1..6] = userId, bidPrice, bidTimeMs, increment, bidId, requestId

if redis.call('SETNX', KEYS[3], ARGV[5]) == 0 then return -1 end
redis.call('EXPIRE', KEYS[3], 300)

local current = tonumber(redis.call('GET', KEYS[1]) or '0')
local newPrice = tonumber(ARGV[2])
local increment = tonumber(ARGV[4])

if newPrice < current + increment then
    redis.call('DEL', KEYS[3])
    return 0
end

redis.call('SET', KEYS[1], newPrice)
local payload = ARGV[5]..'|'..ARGV[1]..'|'..newPrice..'|'..ARGV[3]..'|'..ARGV[6]
redis.call('LPUSH', KEYS[2], payload)
redis.call('LTRIM', KEYS[2], 0, 999)
return 1
```

### 1.4 主流程

```text
1. AOP:鉴权、限流(@RateLimit)、幂等(@Idempotent)
2. 业务校验:商品状态、信用分、保证金、不能给自己出价
3. EVAL bid.lua → 1 成功 / 0 出价不足 / -1 重复
4. 成功 → MQ 发消息(持久化、被超价通知、ES 同步)
5. WebSocket 广播
6. 反狙击:剩余 < 5 分钟时发延长消息
```

### 1.5 BidServiceImpl 骨架

```java
@Service
@RequiredArgsConstructor
public class BidServiceImpl implements BidService {
    private final StringRedisTemplate redis;
    private final RedisScript<Long> bidScript;
    private final RabbitTemplate rabbit;

    @RateLimit(key = "bid:#{#userId}:#{#itemId}", limit = 1, window = 1)
    @Idempotent(key = "#{#requestId}", expireSeconds = 300)
    public BidResult placeBid(Long itemId, Long userId, BigDecimal price, String requestId) {
        AuctionItemBO item = itemQueryService.getCachedItem(itemId);
        validateBidContext(item, userId);
        userExtSvc.checkCanBid(userId, item.getDeposit());

        Long ret = redis.execute(bidScript,
            List.of(RedisKey.itemPrice(itemId), RedisKey.bidQueue(itemId), RedisKey.idem(requestId)),
            String.valueOf(userId), price.toPlainString(),
            String.valueOf(System.currentTimeMillis()),
            item.getBidIncrement().toPlainString(),
            String.valueOf(idWorker.nextId()), requestId);

        if (ret == 0) throw new BizException(ErrorCode.BID_PRICE_TOO_LOW);
        if (ret == -1) throw new BizException(ErrorCode.IDEMPOTENT_DUPLICATE);

        rabbit.convertAndSend(MqExchange.BID, MqRouting.BID_PERSIST, msg);
        wsPusher.pushBidPlaced(itemId, price, userId);
        return BidResult.success();
    }
}
```

## 三、RabbitMQ 设计

### 3.1 交换机/队列拓扑

| 交换机 | 类型 | 队列 | 用途 |
|-------|-----|------|------|
| `bid.exchange` | Direct | `bid.persist.queue` | 出价持久化 |
| | | `bid.notify.queue` | 被超价通知 |
| | | `bid.es.sync.queue` | ES 同步 |
| `auction.exchange` | Direct | `auction.extend.queue` | 反狙击延长 |
| `auction.delay.exchange` | Direct | `auction.delay.queue`(无消费者,TTL→DLX) | 延迟到期 |
| `auction.dlx.exchange` | Direct | `auction.settle.queue` | 实际结算 |
| `order.delay.exchange` | Direct | `order.delay.queue`(TTL→DLX) | 订单超时延迟 |
| `order.dlx.exchange` | Direct | `order.cancel.queue` | 实际取消 |

### 3.2 死信延迟队列(关键)

```text
发布商品 → 投递消息(TTL=拍卖时长) → auction.delay.queue(无消费者)
   → TTL 到期 → DLX → auction.settle.queue → 结算消费者执行
```

延迟消息**不能用单条 TTL**(队列头阻塞问题),而是**为每个时长创建独立的队列**,或用 `rabbitmq-delayed-message-exchange` 插件。

### 3.3 可靠性三连

1. **生产者 confirm**:`publisher-confirm-type: correlated`,callback 处理失败重发或入死信表。
2. **消息持久化**:Exchange/Queue/Message 都 durable。
3. **消费者手动 ack**:`acknowledge-mode: manual`,处理成功后 `basicAck`,失败 `basicNack(requeue=false)` 进死信。

### 3.4 幂等消费

每条消息带 `messageId`(雪花),消费者 `Redis SETNX msg:consumed:<id>` 去重。

### 3.5 配置示例

```java
@Configuration
public class RabbitConfig {
    @Bean
    public DirectExchange bidExchange() {
        return ExchangeBuilder.directExchange("bid.exchange").durable(true).build();
    }
    @Bean
    public Queue bidPersistQueue() {
        return QueueBuilder.durable("bid.persist.queue").build();
    }
    @Bean
    public Binding bindPersist(Queue bidPersistQueue, DirectExchange bidExchange) {
        return BindingBuilder.bind(bidPersistQueue).to(bidExchange).with("bid.persist");
    }

    // 拍卖延迟到期(死信模式)
    @Bean
    public Queue auctionDelayQueue() {
        return QueueBuilder.durable("auction.delay.queue")
            .deadLetterExchange("auction.dlx.exchange")
            .deadLetterRoutingKey("auction.settle")
            .build();
    }
    @Bean
    public Queue auctionSettleQueue() {
        return QueueBuilder.durable("auction.settle.queue").build();
    }
}
```

## 五、缓存策略

### 5.1 三大经典问题

| 问题 | 现象 | 对策 |
|-----|------|------|
| 击穿 | 热点 key 过期瞬间打爆 DB | 互斥锁重建 / 永不过期(后台刷新) |
| 雪崩 | 大量 key 同时过期 | 过期时间加随机扰动 |
| 穿透 | 查不存在的 key | 布隆过滤器 / 空值缓存(短 TTL) |

### 5.2 读写一致性:Cache Aside Pattern

```text
读:先 cache,miss 后查 DB,回填 cache
写:先更 DB,再删 cache(不是更新 cache)
```

**为什么是删而不是更新**:防止两个写并发导致 cache 与 DB 错位。

**延迟双删**:更新 DB → 删 cache → 等 500ms → 再删一次,兜底"删 cache 后还有读重建出旧值"的窗口。

### 5.3 本项目缓存键

| Key | 数据 | TTL |
|-----|-----|-----|
| `auction:item:<id>` | 商品详情 | 10 min + 随机 |
| `auction:cat:tree` | 分类树 | 1h |
| `auction:price:<id>` | 当前价 | 永不过期(拍卖结束清理) |
| `dict:<typeCode>` | 字典 | 1h |
| `auction:hot` | 热门 ZSet | 5 min |

## 六、AOP 自定义注解

### 6.1 `@Log` 操作日志

```java
@Target(METHOD) @Retention(RUNTIME)
public @interface Log {
    String module();
    String type() default "OTHER";
    String description() default "";
}
```

```java
@Aspect @Component
public class LogAspect {
    @Around("@annotation(log)")
    public Object around(ProceedingJoinPoint pjp, Log log) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = null;
        Throwable error = null;
        try {
            result = pjp.proceed();
            return result;
        } catch (Throwable e) {
            error = e; throw e;
        } finally {
            saveLogAsync(log, pjp, result, error, System.currentTimeMillis() - start);
        }
    }
}
```

### 6.2 `@RateLimit` 限流(令牌桶 Lua)

```java
@Target(METHOD) @Retention(RUNTIME)
public @interface RateLimit {
    String key();        // 支持 SpEL 解析参数
    int limit() default 10;
    int window() default 1;  // 秒
}
```

```lua
-- ratelimit.lua: 滑动窗口
-- KEYS[1]=key  ARGV[1]=limit  ARGV[2]=window
local current = redis.call('INCR', KEYS[1])
if current == 1 then redis.call('EXPIRE', KEYS[1], ARGV[2]) end
if current > tonumber(ARGV[1]) then return 0 end
return 1
```

### 6.3 `@Idempotent` 幂等

```java
@Aspect @Component
public class IdempotentAspect {
    @Around("@annotation(idem)")
    public Object around(ProceedingJoinPoint pjp, Idempotent idem) throws Throwable {
        String key = "idem:" + parseSpEL(idem.key(), pjp);
        Boolean ok = redis.opsForValue().setIfAbsent(key, "1", idem.expireSeconds(), TimeUnit.SECONDS);
        if (Boolean.FALSE.equals(ok)) throw new BizException(ErrorCode.IDEMPOTENT_DUPLICATE);
        return pjp.proceed();
    }
}
```

## 七、分布式锁(Redisson)

### 7.1 何时用 Redisson 而非 Lua

- 业务流程复杂,锁内有多步 IO(查 DB、调 MQ),Lua 不合适。
- 需要可重入(同一线程多次 acquire)。
- 需要看门狗自动续期(防业务超过锁默认 30s)。

### 7.2 用法

```java
RLock lock = redisson.getLock("lock:settle:" + itemId);
if (!lock.tryLock(3, 30, TimeUnit.SECONDS)) {
    throw new BizException("正在结算中");
}
try {
    settleAuction(itemId);  // 业务流程
} finally {
    if (lock.isHeldByCurrentThread()) lock.unlock();
}
```

### 7.3 看门狗机制

`tryLock(waitTime, unit)`(不传 leaseTime)启用看门狗,默认 30s,业务执行期每 10s 续期一次。

## 八、限流算法对比

| 算法 | 实现 | 突发 | 平滑 | 选型 |
|-----|-----|-----|-----|------|
| 计数器 | INCR + EXPIRE | 不防 | 否 | 简单接口 |
| 滑动窗口 | ZSet 时间戳 | 防 | 较平滑 | 出价限流 |
| 漏桶 | 队列 + 定速消费 | 强制平滑 | 是 | 出报告等 |
| 令牌桶 | 定速放令牌 + 取走 | 允许突发 | 是 | API 网关 |

本项目:全局滑动窗口 + 出价场景每用户每商品 1秒1次。

## 九、幂等设计

### 9.1 三层防护

1. **客户端**:按钮防抖(disabled 直到响应)、生成 UUID。
2. **接口层**:`@Idempotent` AOP + Header `X-Idempotent-Key`。
3. **业务层**:数据库唯一索引(`biz_bid.client_request_id`)兜底。

### 9.2 幂等 vs 重试

- 幂等:同一请求多次执行 = 一次。
- 重试:失败时再发(MQ 重试、客户端重试)。
- 两者必须配合:能重试是因为幂等。

## 十、事务边界与最终一致

### 10.1 三个事务边界

1. **本地事务**:订单创建(insert order + update item + insert deposit)在一个 `@Transactional` 中。
2. **跨服务最终一致**:本地消息表模式
   - 业务表 + 消息表同事务写。
   - 定时任务扫描"未投递"消息,投递到 MQ。
   - MQ 消费成功标记"已投递"。
3. **补偿对账**:每天凌晨对比 Redis 当前价 / MySQL bid 表 / order 表三方一致性,异常告警。

### 10.2 反狙击延长(典型最终一致)

```text
出价成功(主流程) → MQ 发延长消息(独立事务)
  → 消费者:
     - 加 Redisson 锁
     - SELECT item FOR UPDATE
     - 若 endTime - now < 5min,UPDATE end_time = now + 5min
     - 重置延迟到期消息(撤销旧、发新)
     - 发 WebSocket "endTime 变化"广播
```

## 十一、雪花 ID

避免 UUID 散列写入,采用雪花算法:`64 位 = 1 符号 + 41 时间 + 10 机器 + 12 序列`。

```java
@Component
public class SnowflakeIdWorker {
    private final long workerId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public synchronized long nextId() {
        long ts = System.currentTimeMillis();
        if (ts == lastTimestamp) {
            sequence = (sequence + 1) & 0xFFF;
            if (sequence == 0) ts = waitNextMs(ts);
        } else sequence = 0L;
        lastTimestamp = ts;
        return ((ts - 1577808000000L) << 22) | (workerId << 12) | sequence;
    }
}
```
