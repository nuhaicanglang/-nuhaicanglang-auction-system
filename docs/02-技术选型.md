# 02 · 技术选型

> 选型不是"罗列技术",而是**为每个业务问题挑一把最合适的刀**。
> 这份文档的核心准则:**有需求才上技术,没需求绝不堆砌**。

---

## 一、选型方法论

每选一项技术,我都用三问:

1. **解决了什么具体问题?**(指向需求分析中某一条)
2. **不用它会怎样?**(说明痛点)
3. **替代方案是什么?为什么没选?**(说明权衡)

只有回答得出这三问,才算"想清楚了"。论文答辩、技术面试,问的也都是这三问。

## 三、前端技术栈

### 3.1 Vue 3 + Composition API

- **解决**:构建单页应用(SPA),前后端分离。
- **不用会怎样**:用 jQuery 操作 DOM,代码可维护性极差;用模板引擎服务端渲染(JSP),与 Spring Boot 单体耦合,前端无法独立部署。
- **为什么 Vue 3 不是 Vue 2**:Composition API 更适合复杂业务逻辑复用;TypeScript 一等支持;包体积更小。
- **为什么不是 React**:Vue 中文社区更友好,Element Plus 组件库与中后台业务契合度高,学习曲线对学生更平缓。

### 3.2 Vite

- **解决**:开发时热更新慢的问题。
- **不用会怎样**:Webpack 启动 30 秒,改一行代码等 5 秒,体验差。
- **替代**:Webpack。Vite 基于 ESM 原生模块,冷启动 1 秒内,DX 好。

### 3.3 Pinia(状态管理)

- **解决**:跨组件共享状态(用户登录信息、未读消息数、购物车)。
- **不用会怎样**:每个组件自己存,数据不一致。
- **替代**:Vuex。Pinia 是 Vue 官方"下一代 Vuex",API 更简洁,TypeScript 友好。

### 3.4 Vue Router

- **解决**:SPA 的路由切换、动态路由(根据用户角色加载菜单)、路由守卫(未登录跳转)。

### 3.5 Element Plus

- **解决**:组件库(表单、表格、对话框、上传等),开箱即用,样式统一。
- **替代**:Ant Design Vue、Naive UI。Element Plus 中后台业务最成熟。

### 3.6 Axios + 拦截器

- **解决**:HTTP 请求封装、统一加 Token、统一错误处理。
- **关键设计**:
  - 请求拦截器:自动加 `Authorization: Bearer xxx`。
  - 响应拦截器:统一处理 401(token 过期 → 跳登录)、500、业务错误码。
  - **静默刷新 Token**:发现 401 时用 Refresh Token 换新 Token,重发原请求,用户无感。

### 3.7 其他前端工具

| 工具 | 作用 |
|------|------|
| TypeScript | 类型安全,大型项目必备 |
| ESLint + Prettier | 代码规范统一 |
| dayjs | 时间格式化(替代 moment) |
| WangEditor / Tinymce | 商品详情富文本编辑 |
| ECharts | 后台统计图表 |
| sockjs-client + stompjs | WebSocket 客户端(对接 Spring 的 STOMP) |
| nprogress | 路由切换的顶部进度条 |

## 四、后端核心技术栈

### 4.1 Spring Boot 3.x + Java 17

- **解决**:Web 容器、IoC、AOP、配置管理、自动装配,开箱即用。
- **不用会怎样**:用裸 Servlet 写 Web 应用,自己管理生命周期、依赖注入,代码量爆炸。
- **为什么 Java 17**:LTS 版本;`Record`、`var`、`switch 表达式`、密封类等现代语法;ZGC 默认可用。
- **为什么 Spring Boot 3 不是 2**:Jakarta EE 9+(`javax.*` → `jakarta.*`)是不可逆趋势;原生镜像(GraalVM)支持;长期维护。

### 4.2 Spring Security + JWT

- **解决**:认证(你是谁) + 授权(你能做什么)。
- **不用会怎样**:每个 Controller 自己判断登录状态,代码重复且容易漏。
- **JWT 解决什么**:无状态认证。服务端不用存 session,水平扩展时(同一用户的请求被打到不同实例)依然认得出他是谁。
- **JWT 的坑与对策**:
  - **坑 1**:Token 一旦签发无法吊销 → 对策:Redis 存"黑名单",登出时把 Token 加进去,直到过期。
  - **坑 2**:Token 过长(几百字节,每次请求都带) → 对策:只放必要信息(userId、角色),不放权限明细;权限明细每次从 Redis 查。
  - **坑 3**:Token 续签 → 对策:Access Token(2 小时) + Refresh Token(7 天)双 Token 机制。
- **替代**:Session-Cookie。会话保持需要粘性会话或共享 Session 存储,不利于扩展。

### 4.3 MyBatis-Plus

- **解决**:持久层。MyBatis 的"加强版",省去 90% 的 CRUD 模板代码。
- **不用会怎样**:写 100 个 Mapper XML 写到吐;用纯 JPA 在复杂查询(尤其多表 join、聚合)上力不从心。
- **关键功能**:
  - 通用 Mapper:`save`/`updateById`/`removeById`/`page` 不用写 SQL。
  - 条件构造器(`LambdaQueryWrapper`):类型安全的动态 SQL。
  - 分页插件:`IPage`,自动生成 `LIMIT`。
  - 逻辑删除:`@TableLogic`,SQL 自动加 `deleted=0` 过滤。
  - 自动填充:`@TableField(fill=...)`,创建时间、更新时间、创建人自动填。
  - 乐观锁:`@Version` 字段,`UPDATE` 自动加 `WHERE version=?`。
- **保留 MyBatis 的能力**:复杂 SQL 还是写 XML,这是它比 JPA 强的地方。
- **替代**:Spring Data JPA(简单场景方便,复杂场景痛苦);裸 MyBatis(模板代码多)。

### 4.4 Spring Validation(JSR-303)

- **解决**:参数校验。`@NotNull`、`@Size`、`@Email`、`@Pattern`,声明式。
- **不用会怎样**:每个 Controller 写一堆 `if (xxx == null) throw ...`。
- **进阶**:自定义注解(如 `@PriceRange`),实现 `ConstraintValidator`。

### 4.5 Lombok + MapStruct

- **Lombok**:`@Data`、`@Builder`、`@Slf4j` 减少模板代码。
- **MapStruct**:DTO ↔ Entity 类型映射,编译期生成代码,性能好,比 BeanUtils 反射快几十倍。
- **为什么需要 DTO**:Entity 是"数据库长什么样",DTO 是"前端要什么"。两者职责分离,Entity 字段变了不影响接口。

### 4.6 Hutool

- **解决**:工具类大杂烩(加密、日期、文件、HTTP、二维码、雪花 ID 等)。
- **不用会怎样**:自己写 100 个 `Utils`,或者依赖 Apache Commons + Guava 拼凑。
- **取舍**:不用全家桶,按需引入子模块(如 `hutool-crypto`)。

### 4.7 Knife4j(Swagger 增强)

- **解决**:API 文档自动生成,前后端联调利器。
- **不用会怎样**:手写 Word 文档,改一次接口忘改文档,前端骂后端。
- **替代**:Springdoc-OpenAPI(Spring Boot 3 推荐),Knife4j 4.x 已基于 Springdoc。

### 4.8 Spring Mail(JavaMailSender)

- **解决**:发送邮件 — 注册邮箱验证码、找回密码链接、中标通知、订单异常告警。
- **不用会怎样**:用裸 `javax.mail.Session` 自己拼 MIME 消息;或调第三方 HTTP API(阿里云邮件推送)增加费用与依赖。
- **使用方式**:
  - QQ/163 邮箱开通 SMTP 授权码,放 `application-prod.yml`(用环境变量注入)。
  - `application.yml` 配 `spring.mail.host/port/username/password`。
  - 注入 `JavaMailSender`,用 `MimeMessageHelper` 写 HTML 模板邮件。
  - 验证码邮件走 `notifyExecutor` 线程池**异步**发送,接口不等待 SMTP。
- **模板**:用 Thymeleaf 邮件模板(`@Bean templateEngine`),与正常前端模板隔离。
- **失败重试**:发送失败 → 写"邮件待发表" 或 投递到 RabbitMQ `mail.retry.queue`,消费者重试 3 次后告警。
- **替代**:Apache Commons Email、阿里云邮件推送。本项目用 Spring Mail,零成本。

### 4.9 Hutool Captcha(图形验证码)

- **解决**:登录失败 ≥ 3 次后强制图形验证码 / 注册前置验证码。
- **使用方式**:
  - `CaptchaUtil.createLineCaptcha(120, 40, 4, 50)` 生成 4 字干扰线图。
  - 后端把验证码答案写 Redis `captcha:<captchaId>`,5 分钟过期。
  - 返回 `{ captchaId, base64 }` 给前端,前端 `<img :src="'data:image/png;base64,' + base64">` 渲染。
- **替代**:谷歌 reCAPTCHA(需翻墙)、滑动拼图(实现复杂)。本项目教学场景用 Hutool 足够。

### 4.10 EasyExcel(管理员数据导出)

- **解决**:导出订单/用户/资金流水为 .xlsx。
- **不用会怎样**:用 POI 直接写,API 笨重;大数据量时内存爆炸。
- **优势**:基于 SAX 流式读写,**百万行也只占几 MB 内存**;注解式 mapping。
- **同步导出**(< 5000 行):接口直接返回二进制流。
- **异步导出**(> 5000 行):走 `exportExecutor` 线程池 → 生成临时文件 → 上传 MinIO → 发站内信下载链接。
- **替代**:Apache POI(直接用太底层)、Jxls(模板复杂)。

## 五、数据存储

### 5.1 MySQL 8.x

- **解决**:核心数据持久化、ACID 事务。
- **为什么 8.x**:窗口函数、CTE、JSON 函数、原子 DDL、Hash Join。我们的统计场景会用到窗口函数。
- **存储引擎**:InnoDB(行锁、事务、外键、MVCC)。
- **字符集**:`utf8mb4`(支持 emoji);排序规则 `utf8mb4_0900_ai_ci`(MySQL 8 默认)。
- **关键配置**:
  - `innodb_buffer_pool_size`:占机器内存 50%~70%。
  - `max_connections`:200(配合连接池)。
  - `slow_query_log = ON`:慢日志阈值 1s。
- **替代**:PostgreSQL 更强大但学校教材以 MySQL 为主,生态更广。

### 5.2 Redis 7.x ⭐ 重点

> 你问"Redis 中要用到哪些?"——这部分一次讲全。

#### 5.2.1 Redis 在本系统的 9 大用途

| # | 用途 | 数据结构 | Key 设计示例 |
|---|------|---------|-------------|
| 1 | **缓存**(商品详情/分类树) | String / Hash | `auction:item:1001` |
| 2 | **当前最高价**(出价核心) | String(数值) | `auction:price:1001` |
| 3 | **分布式锁**(出价互斥) | String + Lua | `lock:auction:1001` |
| 4 | **登录失败计数**(防暴力破解) | String + EXPIRE | `login:fail:user@x.com` |
| 5 | **JWT 黑名单**(登出失效) | String + EXPIRE | `jwt:blacklist:<jti>` |
| 6 | **接口限流**(令牌桶 / 滑动窗口) | List / ZSet / Lua | `rate:bid:userId:itemId` |
| 7 | **热门榜单**(出价数 TOP) | ZSet | `auction:rank:hot` |
| 8 | **去重计数**(浏览数,同一用户 1h 内只算 1) | HyperLogLog | `auction:view:1001:hour` |
| 9 | **延迟队列备选**(配合 Redisson) | ZSet 时间戳 | `delay:order:timeout` |
| + | **WebSocket 在线连接** | Hash | `ws:online:<userId>` |
| + | **搜索历史**(每用户最近 10 条) | List + LTRIM | `search:history:<userId>` |
| + | **保证金冻结记录** | Hash | `deposit:frozen:<userId>` |

#### 5.2.2 关于"Redis 持久化"

- **RDB**:周期性快照,适合**容灾**,数据小、恢复快,但可能丢最后几分钟数据。
- **AOF**:追加每次写命令,适合**强一致**,丢数据少(`appendfsync everysec` 最多丢 1 秒),但文件大、恢复慢。
- **本项目策略**:
  - 缓存用途的数据(商品详情缓存)即使丢失也能从 MySQL 重建,**RDB 即可**。
  - 锁、限流、计数等"短生命周期"数据,丢了无所谓。
  - 关键业务状态(如当前最高价)虽然 Redis 也存,**但 MySQL 才是真相之源**(出价记录入库),Redis 重启可从 MySQL 重建。
  - 因此推荐:**RDB 主 + AOF 关闭**,简单且性能好。

#### 5.2.3 关于"Redis 与多线程"

- Redis **核心命令处理是单线程**(不需要锁,天然原子)——这是它快的根本原因。
- Redis 6.0 引入了**网络 I/O 多线程**(读写 socket、协议解析),命令执行依然单线程。
- **对我们的意义**:
  - `INCR`、`SETNX`、ZADD 等单命令是原子的,不需要加锁。
  - 多个命令组合需要原子性时,用 **Lua 脚本**(在 Redis 端单线程执行,期间不会被其他客户端打断)。
  - 出价场景的"比较 + 更新当前最高价"必须用 Lua,这是高并发安全的关键。

#### 5.2.4 Lua 脚本示例(出价核心)

```lua
-- KEYS[1] = auction:price:<itemId>(当前最高价)
-- KEYS[2] = auction:bid:queue:<itemId>(出价记录待入库队列)
-- ARGV[1] = userId  ARGV[2] = bidPrice  ARGV[3] = bidTime  ARGV[4] = increment
local current = tonumber(redis.call('GET', KEYS[1]) or '0')
local newPrice = tonumber(ARGV[2])
local minIncrement = tonumber(ARGV[4])

if newPrice < current + minIncrement then
    return 0  -- 出价不够,失败
end

redis.call('SET', KEYS[1], newPrice)
redis.call('LPUSH', KEYS[2], ARGV[1] .. '|' .. ARGV[2] .. '|' .. ARGV[3])
return 1  -- 成功
```

> 整段脚本在 Redis 端**原子执行**,100 个并发请求过来,只有最快出到合规价的那一个会成功。

#### 5.2.5 Redisson(Redis 的 Java 客户端 + 分布式工具集)

- 提供**可重入锁**、**读写锁**、**信号量**、**延迟队列**等高级抽象。
- 关键特性:**看门狗(Watch Dog)**,如果业务执行时间超过锁默认 30s,会自动续期,避免锁提前释放。
- 我们用 Redisson 做:
  - 出价分布式锁(尽管 Lua 也能做,Redisson 更安全易用)。
  - 订单创建的幂等锁。
  - 延迟队列(订单超时取消,作为 RabbitMQ 死信队列的备选学习对象)。

### 5.3 Elasticsearch 8.x + IK 分词

- **解决**:商品全文检索、分类聚合、价格区间筛选。
- **不用会怎样**:用 MySQL `LIKE '%关键字%'` 不走索引、慢、不支持中文分词、不支持相关度排序。
- **核心概念**:
  - **倒排索引**:从词到文档的映射,搜索快的根本。
  - **分词**:中文必须用 IK 分词器,`ik_smart`(粗粒度,搜索时用) vs `ik_max_word`(细粒度,索引时用)。
  - **mapping**:相当于表结构定义,字段类型(text/keyword/long/date)。
  - **bool 查询**:`must`(必须) / `should`(任一) / `filter`(必须但不算分,可缓存) / `must_not`(不能)。
- **数据同步策略**(MySQL → ES):
  1. **双写**:Service 层同时写两边——简单,但不一致风险高,**不推荐生产**。
  2. **MQ 异步**:写完 MySQL 发消息,消费者写 ES——本项目采用,简单可靠。
  3. **Canal 监听 binlog**:零侵入,实时性最好——生产首选,本项目作为"进阶"作业可选实现。

### 5.4 RabbitMQ 3.x(主选 MQ)

- **解决**:异步、解耦、削峰、最终一致性。
- **不用会怎样**:出价主流程要等通知发完、要等 ES 同步完,接口慢且易雪崩。
- **核心概念**:
  - **Producer / Exchange / Queue / Consumer**。
  - **4 种交换机**:Direct(精确路由)、Topic(模式匹配)、Fanout(广播)、Headers(按 header)。
  - **可靠性三连**:
    1. **生产者 publisher confirm**:确认消息到达 Broker。
    2. **持久化**:Exchange、Queue、Message 三层都要 durable。
    3. **消费者 ack**:手动 ack,处理失败 nack 重入队列或进死信。
  - **死信队列(DLX)**:消息因 TTL 过期/被拒绝/队列满,进入 DLX。**这是我们做"延迟消息"的关键**。
  - **幂等消费**:消息可能重复,在消费者用 `Redis SETNX(消息ID)` 去重。
- **本项目用途**:
  - 出价成功后:发送"通知前一个最高出价者"消息。
  - 出价成功后:发送"持久化出价记录到 MySQL"消息。
  - 出价成功后:发送"同步到 ES"消息。
  - 拍卖创建时:投递延迟消息(TTL=拍卖时长),到点结算。
  - 订单创建时:投递延迟消息(TTL=30 分钟),超时取消。

### 5.5 Kafka(对比与可选替换)

- **不在主选**,但要理解:
  - Kafka 强项是**超高吞吐、顺序消费、可重复消费**(消费者组 offset),适合**日志、埋点、流处理**。
  - RabbitMQ 强项是**灵活路由、死信、可靠投递、低延迟**,适合**业务消息**。
  - 本项目"业务通知/订单"更适合 RabbitMQ。
  - 如果做"用户行为日志""出价审计流",Kafka 更合适。
- **可选实践**:阶段 5 实现 RabbitMQ,阶段 8 用 Kafka 收集出价审计日志,**对比写在论文里**——这是高分点。

## 六、实时通信:WebSocket(STOMP)

- **解决**:出价后所有围观者实时看到当前最高价、剩余时间。
- **不用会怎样**:前端只能轮询(每 2 秒一次),延迟高、浪费带宽、QPS 大。
- **为什么 STOMP 不是裸 WebSocket**:STOMP 在 WebSocket 上加了"发布/订阅"语义,Spring 原生支持,前端用 `stompjs` 一行订阅,后端 `convertAndSend` 一行推送。
- **架构**:
  - 用户进入商品详情页 → 前端订阅 `/topic/auction/{itemId}`。
  - 后端出价成功 → 向该 topic 推送 `{currentPrice, bidder, time}`。
  - 多实例时:用 Redis Pub/Sub 或 RabbitMQ 作为 Spring Messaging 的 Broker,跨实例广播。
- **本项目阶段**:单实例先用内置 Simple Broker;多实例时切到 Redis 或 RabbitMQ,需求文档已预留扩展点。

## 七、定时任务

- **方案 A:Spring `@Scheduled`** —— 单机简单。
- **方案 B:XXL-Job** —— 分布式,有可视化管理后台。
- **方案 C:Redisson 延迟队列 / RabbitMQ 死信队列** —— 单条消息级延迟,精度高。

**本项目选择**:
- 周期性任务(如每天凌晨清理过期数据):`@Scheduled`。
- 单条业务延迟任务(订单超时、拍卖到期):**RabbitMQ TTL+DLX**(主)+ Redisson 延迟队列(可选实现对比)。
- **不引入 XXL-Job**,因为单体足够。

## 八、AOP(面向切面)的具体落地

> 你问 AOP 用在哪——不是为了用而用,有具体场景。

| 切面 | 注解 | 切入点 | 作用 |
|------|------|--------|------|
| `@Log` | 自定义注解 | 标注的方法 | 记录"谁、什么时候、调用了什么、参数、返回、耗时" |
| `@RateLimit` | 自定义注解 | 高频接口 | 基于 Redis + Lua 实现令牌桶限流 |
| `@Idempotent` | 自定义注解 | 写接口 | 基于 Token + Redis 实现幂等 |
| `@DataScope` | 自定义注解 | 列表查询 | 数据权限过滤(管理员看全部,普通用户只看自己) |
| `@Cacheable` | Spring 内置 | 高频读 | 简单缓存 |
| Web 性能监控 | `@Around` 全局 | 所有 Controller | 统计接口响应时间,慢的告警 |

### AOP 的本质收益:
1. **横切关注点的抽离**——日志、限流、幂等都是"业务无关但每个接口都要"的事。
2. **可声明式开关**——加一行注解就生效,撤销也方便。
3. **可组合**——多个切面叠加,各管一摊。

## 九、设计模式的具体落地

> 设计模式不是"必须用",是"用对地方能让代码优雅"。本项目刻意安排了下列模式:

| 模式 | 用在哪 | 解决了什么 |
|------|--------|----------|
| **策略模式** | 拍卖类型(英式/荷兰式/密封式)的不同竞价规则 | 易扩展新拍卖类型 |
| **策略模式** | 通知方式(站内信/邮件/短信)选择 | 通过配置切换 |
| **工厂模式** | 通知发送器创建 | 隔离创建细节 |
| **模板方法** | 出价校验流程(身份→金额→时间→额度) | 统一流程,子步骤可定制 |
| **责任链** | 出价校验链 | 校验项可灵活组合 |
| **观察者** | 出价成功后的多方通知(WS、MQ、统计) | 主流程不与多方耦合 |
| **状态模式** | 商品/订单状态机 | 合法状态迁移,非法被拦 |
| **建造者** | 复杂查询条件构造 | 链式调用,可读性好 |
| **单例** | Spring Bean 默认 | 节省内存,统一入口 |
| **代理(动态代理)** | AOP、MyBatis Mapper | 增强行为 |
| **适配器** | 支付方式接入(支付宝/微信/模拟) | 接口统一 |
| **装饰器** | 缓存装饰(给 Service 加缓存) | 不修改原类 |

## 十、规范类约定

### 10.1 代码规范
- 阿里巴巴 Java 开发手册(IDEA 装 Alibaba Java Coding Guidelines 插件)。
- 包命名:`com.auction.<module>.<layer>`,如 `com.auction.business.controller`。
- 类名:见名知意,Controller 以 `Controller` 结尾,Service 以 `Service` / `ServiceImpl` 结尾。

### 10.2 接口规范
- RESTful:`GET /api/v1/items`(列表)、`POST /api/v1/items`(新增)、`PUT /api/v1/items/{id}`(改)、`DELETE /api/v1/items/{id}`。
- 路径全小写,多词用 `-`(`/api/v1/auction-items`)。
- 资源用复数。
- 操作动词不进 URL,放方法。
- 统一返回:
```json
{ "code": 0, "msg": "success", "data": { ... }, "traceId": "..." }
```
- 业务错误码统一管理(`ErrorCode` 枚举)。

### 10.3 数据库规范
- 表名小写下划线 `auction_item`、`auction_bid`。
- 每张表必有:`id`、`created_at`、`updated_at`、`deleted`、(部分)`version`。
- 字段加注释,SQL 评审走 review。

### 10.4 Git 规范
- 分支:`main`(生产)、`develop`(开发)、`feature/*`、`fix/*`。

## 十一、部署相关

### 11.1 Nginx
- 反向代理 `/api` → 后端、`/ws` → 后端 WebSocket。
- 静态资源(前端 `dist/`)直接由 Nginx 提供。
- HTTPS(Let's Encrypt 免费证书,或自签证书)。
- Gzip 压缩。
- 限流(`limit_req_zone`)。
- 静态资源加版本(hash)+ 强缓存。

### 11.2 Docker + docker-compose
- 每个组件一个容器:MySQL、Redis、RabbitMQ、ES、Backend、Frontend(其实是 Nginx)。
- 用 `docker-compose.yml` 一键 `up -d` 启动整套。
- 数据卷持久化(MySQL/Redis 数据不丢)。
- 网络隔离:内部组件不暴露端口给宿主机,只 Nginx 的 80/443 对外。

### 11.3 监控(选做)
- **Spring Boot Actuator**:`/actuator/health`、`/actuator/metrics`。
- **Prometheus** 抓 Actuator 数据,**Grafana** 可视化。
- **ELK** 收集日志(可选)。

## 十三、关键决策回顾(论文素材)

把"为什么这么选"写下来,论文里就是"3.2 技术选型论证"一节:

1. **为什么单体而不是微服务**:学校项目规模小,微服务带来的运维成本远大于收益;但**模块化**保留了未来拆分的可能。
2. **为什么 RabbitMQ 而不是 Kafka**:业务消息为主、低延迟、灵活路由、死信队列原生支持。
3. **为什么 Redis + Lua 而不是 Redisson 锁**:Lua 在出价场景下"比较 + 更新 + 入队"一气呵成,比"加锁 → 业务 → 解锁"少两次网络往返,QPS 更高。但 Redisson 锁更通用,我们也会在订单创建的幂等场景使用它。
4. **为什么 ES 而不是只用 MySQL**:LIKE 不走索引,无中文分词,无相关度排序。
5. **为什么 JWT 而不是 Session**:无状态,易扩展;Session 共享要么粘性会话要么共享存储,都是负担。
6. **为什么 Vue 而不是 React**:中文社区、Element Plus、学习曲线对学生友好。
