# 09 · 开发计划与里程碑

> 把"我要做完一个项目"拆成 8 个可交付的里程碑,每个里程碑都有验收标准。
> **每完成一个阶段都是一次"答辩可演示"的状态**,这样你不会在项目最后才发现来不及。

---

## 一、总览

| 阶段 | 主题 | 工期 | 关键产出 |
|-----|------|-----|---------|
| 0 | 准备 | 1 天 | 工具链 + 中间件 + 工程骨架 |
| 1 | 用户与权限 | 5 天 | 注册/登录/RBAC/JWT/AOP 日志 |
| 2 | 商品与分类 | 4 天 | CRUD + 文件上传 + 分类树缓存 |
| 3 | 出价核心 | 6 天 | Lua 出价 + 限流 + 幂等 + WS 推送 |
| 4 | MQ 异步流 | 4 天 | 持久化、通知、延迟队列结算 |
| 5 | 订单/保证金 | 4 天 | 订单状态机、保证金账本 |
| 6 | 搜索/统计 | 3 天 | ES 同步 + 搜索 + 统计图表 |
| 7 | 联调/优化/上线 | 4 天 | 压测 + 容灾 + 部署 |
| 论文 | 论文撰写 | 5 天(并行) | 设计文档 → 论文章节 |
| **合计** | **34 天**(约 1 个月)——原 31 天 + 3 天(钱包审计/信用体系/邮箱验证码/导出) | | |

> 你可以根据自己的可投入时间(每天 4h?8h?)调整。这份计划按"每天 6~8 小时" 估。

## 十一、每日工作节奏建议

```text
上午(2~3h):
  1. 看任务清单,确定今天要交付什么
  2. 开 IDE,写代码
  3. 出现问题:先文档化(progress.txt 记一下),再尝试解决
下午(2~3h):
  1. 继续编码
  2. 最后 30 分钟:跑一遍今天写的功能(Postman/前端),确认能用
晚上(1h):
  1. git commit + push,提交信息写清楚
  2. 把今天遇到的"坑"写进项目根 lessons-learned.md
  3. 排明天的 1 件最重要的事
```

## 十二、风险与应对

| 风险 | 应对 |
|-----|------|
| 时间不够 | 优先保证 阶段 1~5,搜索/统计/上线可降级 |
| 某项技术陷入泥潭 | **2 小时不出结果** → 写下问题 → 转其他任务 → 晚上集中查 → 第二天再定 |
| Redis Lua 调不通 | 退化用 Redisson 锁 + 数据库乐观锁 |
| ES 同步出问题 | 退化用 MySQL LIKE,论文里说"未来工作" |
| Docker 跑不起来 | 退化用本地直接跑,部署章节弱化 |
| 论文写不完 | 设计文档已经是论文 70% 素材,直接复用 |
