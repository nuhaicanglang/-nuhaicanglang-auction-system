# Day 29 优化与容灾复核

> 本文记录 Day 29 的慢 SQL 复核、缓存策略复核、关键依赖故障演练方案。当前交付以“可执行检查清单 + 演练步骤”为主，避免在未确认完整环境状态时直接停止 Redis/RabbitMQ。

## 1. 目标

Day 29 关注三件事：

- 慢 SQL 整改：确认核心分页/列表/流水查询都有合适索引，避免全表扫描。
- 缓存策略复核：确认 Redis Key、TTL、失效策略和降级预期清晰。
- 关键依赖故障演练：模拟 Redis、RabbitMQ 不可用时，明确系统表现、恢复步骤和风险边界。

## 2. 慢 SQL 复核

### 2.1 复核脚本

新增脚本：

```text
test/ops/day29-sql-review.sql
```

执行方式：

```powershell
docker exec -i auction-mysql mysql -uroot -proot123456 auction < test/ops/day29-sql-review.sql
```

该脚本只执行：

- `EXPLAIN`
- `SHOW INDEX`

不会修改表结构和业务数据。

### 2.2 核心查询与索引对应关系

| 场景 | 代表查询 | 已有索引 | 结论 |
|---|---|---|---|
| 商品列表：状态 + 创建时间 | `status = ? ORDER BY created_at DESC` | `idx_status_created(status, created_at)` | 可命中 |
| 商品列表：状态 + 结束时间 | `status = ? ORDER BY end_time` | `idx_status_endtime(status, end_time)` | 可命中 |
| 商品列表：分类筛选 | `category_id = ?` | `idx_category(category_id)` | 可命中 |
| 商品列表：标题模糊 | `title LIKE '%xx%'` | 无法有效利用 BTree | 应走 ES 搜索 |
| 出价列表 | `item_id = ? ORDER BY bid_time DESC` | `idx_item_time(item_id, bid_time)` | 可命中 |
| 最高出价 | `item_id = ? ORDER BY bid_price DESC` | `idx_item_price(item_id, bid_price)` | 可命中 |
| 买家订单 | `buyer_id + status + created_at` | `idx_buyer_status_created` | 可命中 |
| 卖家订单 | `seller_id + status + created_at` | `idx_seller_status_created` | 可命中 |
| 钱包流水 | `user_id ORDER BY created_at DESC` | `idx_user_created` | 可命中 |
| 信用日志 | `user_id ORDER BY created_at DESC` | `idx_user_created` | 可命中 |
| 支付超时关闭 | `status + pay_deadline` | `idx_pay_deadline(status, pay_deadline)` | 可命中 |

### 2.3 需要重点观察的查询

#### 商品列表按当前价排序

接口：

```http
GET /api/items?status=3&sort=current_price
```

当前表中有 `idx_status_created` 和 `idx_status_endtime`，没有 `status + current_price` 组合索引。

如果压测中发现该查询 P95/P99 较高，可增加：

```sql
ALTER TABLE biz_auction_item
ADD INDEX idx_status_price (status, current_price);
```

#### 商品列表按出价数排序

接口：

```http
GET /api/items?status=3&sort=bid_count
```

如果热门商品列表或用户列表页大量使用该排序，可增加：

```sql
ALTER TABLE biz_auction_item
ADD INDEX idx_status_bid_count (status, bid_count);
```

#### 标题模糊查询

接口：

```http
GET /api/items?keyword=国画
```

`LIKE '%国画%'` 无法有效使用普通 BTree 索引，大数据量下应改走：

```http
GET /api/search/items?keyword=国画
```

即使用 Elasticsearch 承担全文检索。

## 3. 缓存策略复核

### 3.1 Redis Key 清单

| Key | 用途 | 写入时机 | 失效策略 | 风险 |
|---|---|---|---|---|
| `auction:price:{itemId}` | 商品当前价 | 启动预热、出价 Lua | 当前未统一 TTL | Redis 丢失会影响出价 |
| `auction:status:{itemId}` | 商品当前状态 | 出价/一口价流程 | 当前未统一 TTL | 状态丢失可能导致 Lua 判断异常 |
| `auction:bid:queue:{itemId}` | 出价队列 | Lua `LPUSH` | 消费后逐步清理 | 积压需监控 |
| `auction:idem:{requestId}` | 出价幂等 | Lua 脚本 | 应设置短 TTL | TTL 过长浪费内存，过短可能重复 |
| `auction:item:{itemId}` | 商品信息缓存 | 预留/可扩展 | 应设置 TTL + 变更删除 | 与 DB 一致性风险 |
| `cache:category:tree` | 分类树 | 查询未命中、分类变更 | 24 小时 + 主动删除 | Redis 故障时查 DB |
| `credit:daily:{userId}:{date}` | 信用每日加分限制 | 信用事件应用 | 按天过期 | Redis 故障时限流失效 |
| `search:history:{userId}` | 搜索历史 | 用户搜索 | 长度限制 20 | Redis 故障时历史不可用 |

### 3.2 当前缓存模式

| 模块 | 模式 | 说明 |
|---|---|---|
| 分类树 | Cache-Aside | 先查 Redis，未命中查 DB 并回写，分类变更主动删除 |
| 出价价格 | Redis 主路径 | Redis Lua 原子判断当前价，MySQL 用于最终审计与持久化 |
| 出价队列 | Redis + MQ | Redis 保证竞价原子性，MQ/消费者异步落库 |
| 搜索历史 | Redis List | `LREM + LPUSH + LTRIM` 去重限长 |
| 信用日限额 | Redis 计数 | 控制每日正向加分上限 |

### 3.3 缓存优化建议

- 对 `auction:price:{itemId}` 和 `auction:status:{itemId}`：建议在商品结束/下架/结算后删除或设置过期时间。
- 对 `auction:idem:{requestId}`：建议确认 Lua 脚本设置合理 TTL，例如 5~30 分钟。
- 对分类树：已经有 24 小时 TTL + 分类变更主动删除，策略合理。
- 对搜索历史：限长 20，内存风险可控。
- 对信用日限额：需要确保 key 到次日自动过期，避免长期堆积。

## 4. Redis 故障演练

### 4.1 演练前提

只建议在开发环境执行，不要在有真实用户请求的环境执行。

确认服务：

```powershell
docker ps
curl http://localhost:8080/api/ping
```

### 4.2 停 Redis

```powershell
docker stop auction-redis
```

### 4.3 观察点

| 功能 | 预期表现 |
|---|---|
| `/api/categories/tree` | 如果无降级，可能出现 Redis 连接异常；后续可优化为 Redis 异常时直接查 DB |
| 出价接口 | 应失败，因为 Redis Lua 是竞价强一致核心依赖 |
| 搜索历史 | 应失败或不可用，但不影响匿名搜索本身 |
| 信用每日限额 | 信用事件可能失败或限额失效，需依赖日志排查 |
| 后端进程 | 不应崩溃退出 |

### 4.4 恢复 Redis

```powershell
docker start auction-redis
```

恢复后建议：

- 重启后端，让 `AuctionCacheWarmUp` 重新预热拍卖中商品价格。
- 或手动触发未来的缓存重建接口/任务。

## 5. RabbitMQ 故障演练

### 5.1 停 RabbitMQ

```powershell
docker stop auction-rabbitmq
```

### 5.2 观察点

| 功能 | 预期表现 |
|---|---|
| 出价接口 | Redis Lua 可能成功，但发送 `bid.persist` MQ 失败时接口应报错；需要关注 Redis 价格已变但 MQ 未落库的风险 |
| 商品审核通过 | ES 同步消息发送失败，商品搜索索引可能延迟/缺失 |
| 拍卖结算 | 延迟队列不可用，结算消息无法投递或消费 |
| 订单超时关闭 | 延迟关闭消息不可用，订单可能不会自动关闭 |
| 信用事件 | 信用 MQ 不可用，信用分可能延迟更新 |
| 后端进程 | 不应崩溃退出，应记录明确错误日志 |

### 5.3 恢复 RabbitMQ

```powershell
docker start auction-rabbitmq
```

恢复后检查：

```powershell
# 管理台
# http://localhost:15672
# admin / admin123
```

重点查看：

- 队列是否恢复声明
- 是否有消息积压
- 消费者是否重新连接
- 后端日志是否出现 RabbitMQ 重连成功信息

## 6. 现阶段风险结论

### 6.1 已具备的可靠性设计

- RabbitMQ 队列、交换机持久化。
- RabbitTemplate 设置 `mandatory=true`，无法路由会回退。
- ConfirmCallback 可记录交换机投递失败。
- 出价落库消费者使用 `client_request_id` 唯一索引做幂等。
- 拍卖结算、订单超时均使用延迟队列，消费者具备重复/过期校验思想。
- ES 搜索异常时已有空结果兜底。

### 6.2 后续可优化项

- Redis 异常时，分类树等非强依赖功能可降级查 DB。
- MQ 发送失败时，关键业务需要本地消息表或 Outbox 模式做补偿。
- 商品状态/价格 Redis Key 在结算/下架后应统一清理。
- 为 `status + current_price`、`status + bid_count` 等高频排序建立组合索引，需要以 Day28 实测结果决定。
- 将 RabbitMQ 消费失败的消息接入死信队列和人工补偿台。
