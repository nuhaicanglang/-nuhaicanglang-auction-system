# 05 · 接口设计(REST API)

> 接口是前后端、外部系统对系统的"操作面板"。
> 这份文档要做到:**任何一名前端工程师拿着它就能开工,而不需要再问任何问题**。

---

## 一、设计原则

### 1.1 RESTful 风格

| HTTP 方法 | 语义 | 示例 |
|----------|------|------|
| `GET`    | 查询(幂等、可缓存) | `GET /items/123` |
| `POST`   | 创建/动作 | `POST /items`、`POST /bids` |
| `PUT`    | 全量更新(幂等) | `PUT /items/123` |
| `PATCH`  | 部分更新 | `PATCH /items/123` |
| `DELETE` | 删除(幂等) | `DELETE /items/123` |

### 1.2 URL 规范

- 全小写,多词用 `-`(`/auction-items`)。
- 资源用复数名词,**不用动词**:
  - ✅ `POST /items` 创建
  - ✅ `POST /items/123/bids` 出价(在 item 资源下创建一个 bid 子资源)
  - ❌ `POST /createItem`
- 路径里只放资源标识,过滤、分页、排序进 query string:
  - `GET /items?categoryId=110&status=3&page=1&size=20&sort=-end_time`
- 一级路径前缀 `/api/v1/...`,版本号写在 URL,便于灰度。
- 管理端接口前缀 `/api/v1/admin/...`,与用户端隔离。

### 1.3 安全规范

- 所有非公开接口必须带 JWT:`Authorization: Bearer <token>`。
- 敏感操作(出价、支付、密码修改)在 Header 中携带 `X-Idempotent-Key` 实现幂等。
- 所有写接口添加 CSRF 防护(JWT 模式天然 OK,Cookie 模式需 SameSite)。
- 文件上传走专用 `/api/v1/upload`,大小限制 5MB/张,30MB/总。

## 二、统一返回格式

### 2.1 响应体结构

```json
{
  "code": 0,
  "msg": "success",
  "data": { ... },
  "traceId": "9b8c7d6e5f4a3b2c1"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | `0` 表示成功;非 `0` 表示业务错误码 |
| `msg` | string | 错误描述,可直接展示给用户 |
| `data` | any | 业务数据 |
| `traceId` | string | 链路追踪 ID,排查问题时给后端 |

### 2.2 分页返回结构(`data` 内部)

```json
{
  "code": 0,
  "data": {
    "total": 1234,
    "pageNum": 1,
    "pageSize": 20,
    "list": [ {...}, {...} ]
  }
}
```

### 2.3 HTTP 状态码与业务错误码并存

- HTTP 状态码:遵循 HTTP 语义(200/400/401/403/404/500)。
- 业务错误码:在 `code` 字段返回,**HTTP 200 也可能 `code != 0`**(业务失败,但通信成功)。
- 仅在以下情况用 4xx/5xx HTTP 状态码:
  - 401:未登录或 Token 失效。
  - 403:权限不足。
  - 404:URL 不存在(框架级)。
  - 405:方法不允许。
  - 500:未捕获的服务端异常。

## 四、分页与排序约定

### 4.1 入参

```text
?page=1&size=20&sort=-end_time,price
```

- `page`:从 1 开始,默认 1。
- `size`:默认 20,最大 100。
- `sort`:多字段逗号分隔,字段名前加 `-` 表示倒序。

### 4.2 后端实现要点

- MyBatis-Plus 分页插件:`Page<T> page = new Page<>(pageNum, pageSize);`
- 安全考虑:**`sort` 字段必须白名单校验**,防 SQL 注入。

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|

| POST | `/auth/logout` | 登出(token 加黑名单) | 🔒 |

| POST | `/auth/email-code/send` | 发送邮箱验证码(需先过图形码);入参：`{ email, scene, captchaId, captchaCode }`;`scene` 枚举：`REGISTER`/`CHANGE_EMAIL` | 🌐 |
| POST | `/auth/password/change` | 修改密码 | 🔒 |

#### 5.1.0 注册请求样例

```http
POST /api/v1/auth/register
Content-Type: application/json

{
  "username":    "alice",
  "password":    "Abc12345!",
  "email":       "alice@example.com",
  "captchaId":   "uuid-xxx",
  "captchaCode": "5G7K",
  "emailCode":   "846219"
}
```

后端校验顺序：图形码 → 邮箱码(Redis `email:code:REGISTER:<email>`) → 邮箱/用户名唯一性 → 插 user + user_extension(`balance=0, credit=80`)。

#### 5.1.2 验证码限频与防刷

- 同邮箱：60 秒内不能重发(`email:cooldown:<email>`)
- 同邮箱：每天最多 10 次(`email:daily:<email>`)
- 同 IP：每小时最多 30 次(`email:ip:<ip>`)
- 超限返回 `90001` 限流

## 七、接口幂等性约定

### 7.1 哪些接口需要幂等

- 出价(`POST /items/{id}/bids`)
- 创建订单(系统内部触发,但模拟支付时也需要)
- 支付(`POST /orders/{id}/pay`)
- 重要状态变更(发货、确认收货、取消)

### 7.2 实现方式

- 客户端生成 UUID,放在 Header `X-Idempotent-Key`。
- 后端 `@Idempotent` AOP:Redis SETNX `idem:<userId>:<key>`,过期 5 分钟。
- 重复请求直接返回上次结果或拒绝。

## 八、文档自动生成与联调

### 8.1 Knife4j

- 引入 `knife4j-openapi3-jakarta-spring-boot-starter`。
- Controller 加 `@Tag`、`@Operation`、`@Parameter`、`@Schema` 注解。
- 访问 `http://localhost:8080/api/doc.html` 在线调试。

### 8.2 Postman 集合

- 上线后导出 OpenAPI JSON,导入 Postman 给前端用。
- 环境变量:`{{baseUrl}}`、`{{token}}`。

## 九、版本演进策略

- 兼容性优先:旧接口不删,新接口加在 `/api/v2/`。
- 字段添加可在原 DTO 加;字段删除/语义变化必须升版。
- 灰度发布:Nginx 按用户灰度路由 v1/v2。

## 十、限流与降级返回约定

- 触发限流时返回:
```json
{ "code": 90001, "msg": "请求过于频繁,请稍后再试", "data": null }
```
- 触发熔断/降级时返回:
```json
{ "code": 90002, "msg": "服务繁忙,请稍后再试", "data": null }
```
- 前端可全局拦截这两种 code,弹个统一提示。
