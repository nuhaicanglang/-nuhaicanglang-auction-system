# 01 · 需求分析

> 任何一个工程的失败,99% 不是技术不行,而是需求没搞清楚。
> 这份文档要回答四个根本问题:**给谁用?用来做什么?有哪些约束?成功的标准?**

---

## 一、业务背景与目标

### 1.1 背景

传统线下拍卖效率低、覆盖范围有限;现有线上拍卖平台(eBay 海外、阿里拍卖、京东拍卖)证明了商业模式成熟。本项目模拟此类平台,聚焦**英式拍卖**(价高者得,公开出价)。

### 1.2 业务目标

1. 卖家可以**便捷发布**拍卖商品(起拍价、加价幅度、一口价、拍卖时长)。
2. 买家可以**安全、实时**地对感兴趣的商品出价。
3. 系统**自动**处理结算、订单、通知,减少人工。
4. 平台具备**搜索、筛选、推荐**能力,提升撮合效率。
5. 数据可统计、可分析,为运营决策提供依据。

### 1.3 非业务目标(技术训练目标)

- 学会"高并发场景下的数据一致性"工程实现。
- 学会构建"前后端分离 + 多模块后端"的标准工程。
- 学会容器化部署、压测、性能调优。

## 二、用户角色与权限

### 2.1 角色定义

| 角色 | 描述 | 主要诉求 |
|------|------|---------|
| **游客** (Guest) | 未登录用户 | 浏览商品、查看详情、看公开榜单 |
| **注册用户** (User) | 完成注册的人,默认同时是买家和卖家 | 出价、发布商品、管理个人订单 |
| **卖家** (Seller, 角色态) | 当用户发布商品时进入此态 | 管理自己的拍品、查看竞价情况 |
| **买家** (Buyer, 角色态) | 当用户出价时进入此态 | 出价、收藏、查看出价记录 |
| **管理员** (Admin) | 平台运营人员 | 用户/商品/分类管理、违规处理、统计 |
| **超级管理员** (Super Admin) | 拥有所有权限,可分配管理员 | 系统配置、权限分配 |

> 设计上,买家和卖家**不是独立账户**,而是同一个 `User` 在不同业务上下文下的"角色态"。这符合实际(淘宝就是这样),也降低了系统复杂度。如果后续要做"企业卖家认证",可以扩展一个 `seller_profile` 表。

### 2.2 权限矩阵(节选)

权限按"模块:操作"命名,例如 `auction:bid` 表示"拍卖:出价"。完整权限码列表见 `04-数据库设计.md` 中的 `sys_permission` 表。

| 功能 | 游客 | 用户 | 管理员 | 超管 |
|------|:---:|:---:|:---:|:---:|
| 浏览商品列表 | ✅ | ✅ | ✅ | ✅ |
| 查看商品详情 | ✅ | ✅ | ✅ | ✅ |
| 注册/登录    | ✅ | - | - | - |
| 发布拍卖商品 | ❌ | ✅ | ✅ | ✅ |
| 出价         | ❌ | ✅ | ❌(防内幕) | ❌ |
| 修改自己的密码 | ❌ | ✅ | ✅ | ✅ |
| 撤销出价(限规则内) | ❌ | ✅ | - | - |
| 强制下架商品 | ❌ | ❌ | ✅ | ✅ |
| 用户禁言/封号 | ❌ | ❌ | ✅ | ✅ |
| 商品分类管理 | ❌ | ❌ | ✅ | ✅ |
| 系统配置(拍卖规则) | ❌ | ❌ | ❌ | ✅ |
| 角色权限分配 | ❌ | ❌ | ❌ | ✅ |
| 数据统计与报表 | ❌ | ❌ | ✅ | ✅ |

> 把"管理员不能出价"明确写下来,这是**业务风控**:防止平台内部人员利用信息差出价。

## 三、核心业务流程

### 3.1 主流程:发布 → 出价 → 成交

```text
卖家 ─发布商品─► 系统(状态:待审核/直接上架)
                  │
                  ▼
              系统进入"拍卖中"
                  │
                  ▼ (买家围观与出价)
            ┌─────┴─────┐
            │           │
       到达一口价     拍卖到点
            │           │
            ▼           ▼
       立即成交    最高价者中标
            │           │
            └─────┬─────┘
                  ▼
          系统生成订单(待支付)
                  │
                  ▼
          买家30分钟内支付
            │           │
       支付成功      超时未付
            │           │
            ▼           ▼
       订单完成     订单取消、保证金没收(部分赔卖家)
            │           │
            ▼           ▼
       卖家发货      商品重新上架/下架
            │
            ▼
       买家确认收货
            │
            ▼
       双方互评 → 信用分变化
```

### 3.2 出价子流程(技术核心)

```text
用户点"出价" ──► 输入金额 ──► 前端校验 ──► 调用接口
                                              │
                                              ▼
                                  ┌─── Redis 分布式锁(商品ID) ───┐
                                  │                                │
                                  │   1. 校验商品状态(拍卖中)    │
                                  │   2. 校验用户(已登录、非卖家、有保证金)│
                                  │   3. 校验金额(>当前最高+加价幅度) │
                                  │   4. 更新 Redis 当前最高价       │
                                  │   5. 写出价记录到 MQ            │
                                  │                                │
                                  └────── 释放锁 ───────────────┘
                                              │
                                              ▼
                                       返回成功
                                              │
                                              ▼
                              WebSocket 推送给所有围观者
                                              │
                                              ▼
                              MQ 消费:把出价记录持久化到 MySQL
                                              │
                                              ▼
                              MQ 消费:通知"前一个最高出价者"被超价
                                              │
                                              ▼
                              如果是一口价 → 触发立即成交流程
```

### 3.3 拍卖到期结算子流程

```text
拍卖创建时 ──► 投递延迟消息(TTL=拍卖时长)到 MQ
                       │
                       ▼ (拍卖时长后)
              延迟消息进入死信队列
                       │
                       ▼
              结算消费者拿到消息
                       │
                       ▼
              ┌─── 数据库事务 ───┐
              │                    │
              │  1. 锁商品行       │
              │  2. 状态置为"已结束"│
              │  3. 查最高出价者   │
              │  4. 生成订单(待支付)│
              │  5. 释放其他出价者保证金│
              │                    │
              └────────────────────┘
                       │
                       ▼
              MQ:推送中标通知 / 流拍通知
                       │
                       ▼
              MQ:再投一条延迟消息(TTL=30 分钟,订单超时)
```

### 4.10 信用分体系(F-CREDIT) ⭐

#### 4.10.1 加减分规则

| 场景 | 变动 | 限制 |
|------|------|------|
| 买家完成订单(收货 + 评价) | +1 | 每日上限 +5 |
| 卖家完成订单 | +2 | 每日上限 +5 |
| 卖家 48 小时内发货 | +1 | 超过不加 |
| 收到 5 星好评 | +1 | - |
| 月度自然恢复 | +5 | 上限 80(获 VIP 须靠加分) |
| 中标后超时未付 | -10 | - |
| 卖家拒发货/超时未发 | -15 | - |
| 收到 1 星差评 | -3 | - |
| 被管理员撤销出价(违规) | -5 ~ -20 | 按严重程度 |
| 被举报核实 | -20 | - |
| 管理员手动调整 | 任意 | 需填原因,有审计 |

#### 4.10.2 分段权益

| 区间 | 标识 | 权益 / 限制 |
|------|------|------|
| ≥ 90 | 🌟 VIP | 保证金减半;商品可选免审核 |
| 80 ~ 89 | ✓ 正常 | 默认 |
| 60 ~ 79 | ⚠ 警告 | 不能一口价;保证金加倍;出价上限 5 万 |
| 30 ~ 59 | ⛔ 受限 | 禁出价;禁发布商品(只能浏览) |
| < 30 | 🚫 冻结 | 自动禁用账号,管理员介入 |

#### 4.10.3 需求清单

| 编号 | 需求 | 说明 |
|------|------|------|
| F-CREDIT-01 | 初始分 | 新用户 80;可配置 |
| F-CREDIT-02 | 加分触发 | 订单完成/好评等事件 → 发 MQ 消息 → `CreditService` 消费 → 同事务更新主表 + 写流水 |
| F-CREDIT-03 | 减分触发 | 同上 |
| F-CREDIT-04 | 个人查看 | `/me/credit-history` 分页看所有变动记录 |
| F-CREDIT-05 | 管理员查看 | 可查任何用户信用变动 |
| F-CREDIT-06 | 月度恢复 | 定时任务(每月1日 02:00)全量扫描,不到 80 的 +5(上限 80) |
| F-CREDIT-07 | 阈值拦截 | `BidValidator` 中按区间拦截;错误码明确提示原因 |

## 六、非功能性需求

### 6.1 性能指标

| 指标 | 目标 |
|------|------|
| 商品列表 P99 响应 | < 200ms |
| 商品详情 P99 响应 | < 300ms |
| **出价接口 P99 响应** | **< 500ms(关键)** |
| 出价接口 QPS(单机) | ≥ 500 |
| 系统可用性 | 99.5%(允许每月 3.6 小时宕机,练习级) |
| 单商品并发出价 | 100 个用户同时出价不出错 |

### 6.2 安全要求

- 所有密码必须 BCrypt 加盐哈希,**禁止明文/MD5**。
- 所有接口必须 HTTPS(部署阶段)。
- JWT 有效期 2 小时,Refresh Token 7 天。
- 防 SQL 注入(MyBatis 参数化、禁止字符串拼接)。
- 防 XSS(后端对富文本字段使用 jsoup 白名单清洗)。
- 防 CSRF(使用 JWT,前端不依赖 cookie 即可天然免疫,但仍建议加 SameSite)。
- 敏感操作(出价、支付)记录审计日志。
- 出价金额必须用 `BigDecimal`,**禁止 float/double**(浮点误差)。
- 用户列表中其他用户的手机号/邮箱要脱敏(`13****1234`)。
- 接口限流:全局 1000 QPS;敏感接口(出价/登录)按用户限流。

### 6.3 可扩展性

- 模块化设计,业务模块可独立替换(例如把保证金从"冻结"改成"预授权扣款")。
- 配置中心化,业务规则(加价幅度范围、保证金比例、超时时间)可热更新。
- 接口版本化(`/api/v1/...`),为未来兼容老客户端留口子。
- 数据库:订单/出价记录表预留分库分表的设计(用户 ID 取模)。

### 6.4 兼容性

| 项目 | 要求 |
|------|------|
| 浏览器 | Chrome 100+ / Edge 100+ / Firefox 100+;不支持 IE |
| 屏幕 | 桌面端为主,移动端响应式适配(后期可单独做小程序) |
| 后端 JDK | 17+ |
| 数据库 | MySQL 8.0+(必须,因为用到 JSON 函数和窗口函数) |

### 6.5 可维护性

- 代码注释覆盖率 ≥ 30%,核心算法 100% 注释。
- 命名规范:阿里巴巴 Java 开发手册。
- 单元测试覆盖核心业务逻辑(Service 层),覆盖率 ≥ 60%。
- 日志:每次重要操作落库 + 文件;ERROR 必须打全栈;敏感数据(密码、token)不打日志。
- API 文档自动生成(Knife4j),保持与代码同步。

## 七、约束条件

### 7.1 业务约束

- 同一用户同一商品同一秒最多出价 1 次。
- 拍卖时长选项:1h、6h、12h、24h、3d、7d;不允许任意填写。
- 最低起拍价:1 元;最高:1,000,000 元(数据库 decimal(12,2) 范围内)。
- 一次最多上传图片:9 张;单张 ≤ 5MB;总大小 ≤ 30MB。
- 商品标题:5~100 字;描述:10~10000 字。

### 7.2 技术约束

- 部署机器:1 台 Linux(2C4G 起步,练习级)。
- 不引入 Spring Cloud(单体足以,避免过度设计)。
- 不上 K8s(Docker Compose 即可)。
- 数据库:单实例 + 读写分离的设计但不实际部署从库(配置预留)。

### 7.3 合规约束

- 用户实名信息:保留但不展示;符合最小权限原则。
- 拍卖品禁止类目:武器、毒品、文物、人体器官等(系统硬编码黑名单)。

## 十、验收标准(摘录)
