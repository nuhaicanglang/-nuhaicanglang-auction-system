# 04 · 数据库设计

> 这是整个系统的"地基"。地基不稳,后面建什么都得拆。
> 这份文档包含:**ER 关系**、**21 张表完整 DDL**、**索引策略**、**种子数据**、**规约说明**。

---

## 一、设计原则

1. **范式与反范式平衡**:OLTP 主流第三范式;高频查询字段适度冗余(如 `auction_item.bid_count`、`final_price`)。
2. **每张表必有四件套**:`id` / `created_at` / `updated_at` / `deleted`(逻辑删除)。
3. **金额字段一律 `DECIMAL(12,2)`**:禁用 float/double。
4. **状态字段一律 `TINYINT UNSIGNED`** 配合常量类与字典表。
5. **时间字段 `DATETIME`** 而非 `TIMESTAMP`(`TIMESTAMP` 有 2038 问题、时区敏感)。
6. **主键统一雪花 ID(`BIGINT`)**:全局唯一、有序、无 GUID 散列写入痛点。
7. **大文本字段单独拆表**:`auction_item.description`(longtext)放主表,如未来发现写放大可拆 `auction_item_detail`。
8. **表名前缀**:`sys_*`(系统域)、`biz_*`(业务域)。
9. **字符集**:`utf8mb4` / `utf8mb4_0900_ai_ci`。
10. **存储引擎**:全部 InnoDB。

**核心关系简述**:
- `sys_user` ↔ `sys_role` 多对多 (`sys_user_role`)
- `sys_role` ↔ `sys_permission` 多对多 (`sys_role_permission`)
- `sys_user` 1 ↔ 1 `sys_user_extension`(扩展信息单独成表方便扩展)
- `biz_category` 自引用(`parent_id` 形成树)
- `biz_auction_item.seller_id` → `sys_user.id`
- `biz_auction_item.category_id` → `biz_category.id`
- `biz_bid.bidder_id` → `sys_user.id`,`biz_bid.item_id` → `biz_auction_item.id`
- `biz_order.item_id`、`buyer_id`、`seller_id` 三个外键
- `biz_deposit.user_id`、`item_id`
- `biz_favorite.user_id`、`item_id`(联合唯一)
- `biz_review.order_id`、`from_user_id`、`to_user_id`

### 4.2 系统域表

```sql
-- ============================
-- 用户表
-- ============================
DROP TABLE IF EXISTS `sys_user`;
CREATE TABLE `sys_user` (
    `id`              BIGINT UNSIGNED NOT NULL                COMMENT '雪花ID',
    `username`        VARCHAR(50)     NOT NULL                COMMENT '登录名(唯一)',
    `nickname`        VARCHAR(50)     NOT NULL                COMMENT '昵称',
    `password`        VARCHAR(100)    NOT NULL                COMMENT 'BCrypt 密码',
    `email`           VARCHAR(100)    DEFAULT NULL            COMMENT '邮箱',
    `phone`           VARCHAR(20)     DEFAULT NULL            COMMENT '手机号',
    `avatar`          VARCHAR(500)    DEFAULT NULL            COMMENT '头像URL',
    `gender`          TINYINT UNSIGNED DEFAULT 0              COMMENT '0未知/1男/2女',
    `real_name`       VARCHAR(50)     DEFAULT NULL            COMMENT '真实姓名(实名)',
    `id_card_no`      VARCHAR(64)     DEFAULT NULL            COMMENT '身份证号(加密存)',
    `status`          TINYINT UNSIGNED NOT NULL DEFAULT 1     COMMENT '1正常/0禁用/2黑名单',
    `blacklist_reason` VARCHAR(255)   DEFAULT NULL            COMMENT '拉黑原因(status=2 时填)',
    `blacklisted_by`  BIGINT UNSIGNED DEFAULT NULL            COMMENT '拉黑操作人',
    `blacklisted_at`  DATETIME        DEFAULT NULL            COMMENT '拉黑时间',
    `last_login_at`   DATETIME        DEFAULT NULL            COMMENT '最后登录时间',
    `last_login_ip`   VARCHAR(50)     DEFAULT NULL            COMMENT '最后登录IP',
    `tenant_id`       BIGINT UNSIGNED NOT NULL DEFAULT 0      COMMENT '租户ID(预留)',
    `created_by`      BIGINT UNSIGNED DEFAULT NULL            COMMENT '创建人',
    `created_at`      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP                  COMMENT '创建时间',
    `updated_by`      BIGINT UNSIGNED DEFAULT NULL            COMMENT '更新人',
    `updated_at`      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    `deleted`         TINYINT UNSIGNED NOT NULL DEFAULT 0     COMMENT '0未删/1已删',
    PRIMARY KEY (`id`),
    UNIQUE KEY `uk_username` (`username`, `deleted`),
    UNIQUE KEY `uk_email`    (`email`, `deleted`),
    UNIQUE KEY `uk_phone`    (`phone`, `deleted`),
    KEY `idx_status` (`status`),
    KEY `idx_created_at` (`created_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户表';

-- ============================
-- 用户扩展表(分表存放高频变化字段)
-- ============================
DROP TABLE IF EXISTS `sys_user_extension`;
CREATE TABLE `sys_user_extension` (
    `user_id`         BIGINT UNSIGNED NOT NULL                COMMENT '用户ID(同sys_user.id)',
    `credit_score`    INT UNSIGNED    NOT NULL DEFAULT 80     COMMENT '信用分0~100',
    `balance`         DECIMAL(12,2)   NOT NULL DEFAULT 0      COMMENT '账户余额',
    `frozen_amount`   DECIMAL(12,2)   NOT NULL DEFAULT 0      COMMENT '冻结金额(保证金等)',
    `total_bid_count` INT UNSIGNED    NOT NULL DEFAULT 0      COMMENT '出价总次数',
    `total_won_count` INT UNSIGNED    NOT NULL DEFAULT 0      COMMENT '中标总次数',
    `total_sold_count`INT UNSIGNED    NOT NULL DEFAULT 0      COMMENT '成交卖出次数',
    `total_amount`    DECIMAL(14,2)   NOT NULL DEFAULT 0      COMMENT '历史成交总额(买+卖)',
    `version`         INT UNSIGNED    NOT NULL DEFAULT 0      COMMENT '乐观锁',
    `updated_at`      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`user_id`),
    KEY `idx_credit_score` (`credit_score`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户扩展信息';

-- ============================
-- 角色表
-- ============================
DROP TABLE IF EXISTS `sys_role`;
CREATE TABLE `sys_role` (
    `id`              BIGINT UNSIGNED NOT NULL                COMMENT '主键',
    `code`            VARCHAR(50)     NOT NULL                COMMENT '角色编码 USER/ADMIN/SUPER',
    `name`            VARCHAR(50)     NOT NULL                COMMENT '角色名',
    `description`     VARCHAR(255)    DEFAULT NULL            COMMENT '说明',
    `sort_order`      INT             NOT NULL DEFAULT 0      COMMENT '排序',
    `status`          TINYINT UNSIGNED NOT NULL DEFAULT 1     COMMENT '1启用/0停用',

    `updated_at`      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    `deleted`         TINYINT UNSIGNED NOT NULL DEFAULT 0,
```

## 五、索引设计原则与本项目应用

### 5.1 通用原则
1. **主键聚簇**:用雪花 ID(有序),减少 InnoDB 页分裂。
2. **唯一索引**:用于业务唯一约束(用户名、邮箱、订单号、客户端幂等 ID)。
3. **联合索引最左前缀**:把"高选择性""等值过滤"的字段放前。
4. **索引覆盖**:常用查询字段都进索引,避免回表。
5. **避免过多索引**:每个写操作要更新所有索引,InnoDB 写放大严重。
6. **不在索引列上做函数运算**:`WHERE DATE(created_at)='...'` 不走索引。

### 5.2 本项目重点索引解析

#### `biz_auction_item`
- `idx_status_endtime(status, end_time)`:用于"拍卖中且即将结束"的列表查询(`WHERE status=3 ORDER BY end_time ASC`),走索引顺序扫描。
- `idx_endtime(end_time)`:定时任务扫"该结束未结束"的兜底查询。
- `idx_seller`:查"我的拍品"。

#### `biz_bid`
- `idx_item_time(item_id, bid_time DESC)`:商品详情页"最近出价 50 条"。
- `idx_item_price(item_id, bid_price DESC)`:"出价排行"查询。
- `uk_request_id`:幂等去重(同一个客户端请求重复,不创建第二条出价)。
- 这张表是**未来分表的最优先候选**,按 `item_id % N` 分表,所有重要查询都带 `item_id`,不会跨片。

#### `biz_order`
- `uk_item`:防止同一商品两个订单(数据库级保护,业务层也应保证)。
- `idx_pay_expire(status, pay_expire_at)`:扫超时未支付订单。

### 5.3 性能基准(预期)

| 查询 | 数据量 | 目标耗时 |
|-----|--------|---------|
| 商品详情(走主键) | - | < 5ms |
| 商品列表分类筛选 | 10 万商品 | < 30ms |
| 出价记录最近 50 条 | 100 万条 | < 20ms |
| 用户中心订单列表 | - | < 50ms |
| 全文检索(走 ES) | 10 万商品 | < 100ms |

## 十、特别说明:为什么不用外键约束

- **不在 DDL 中声明 `FOREIGN KEY`**,只在 ER 图与文档中标注。
- 原因:外键约束在大表写入时锁开销大、跨实例分库分表后无法生效、阻碍数据迁移。**完整性靠应用层保证**。
- 这是阿里 Java 开发手册的硬性规定,生产标准做法。
