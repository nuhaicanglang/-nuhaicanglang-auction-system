# 00 · 学习路线与项目总览

> 这份文档是"地图"。先看清楚整张地图,再开始走路,你不会迷路。

---

## 一、为什么选"在线拍卖"作为学习载体

很多大作业是 CRUD 后台(图书管理、学生信息),技术上几乎不会接触到生产环境真正的难点。**拍卖不一样**:

| 业务特性 | 对应的技术挑战 | 你会被迫学到 |
|---------|---------------|-------------|
| 多人同时出价 | 高并发、竞态条件 | 分布式锁、乐观锁、CAS、Lua 原子脚本 |
| 出价不能丢、不能错 | 数据一致性 | 事务、幂等、消息可靠投递 |
| 实时看到别人出价 | 实时通信 | WebSocket、Server-Sent Events |
| 拍卖到点自动结束 | 定时调度、延迟任务 | XXL-Job、Redis 延迟队列、RabbitMQ 死信 |
| 商品搜索 | 全文检索 | Elasticsearch、IK 分词 |
| 出价高峰期不能崩 | 削峰、限流、缓存 | MQ、Redis 缓存、Sentinel/Resilience4j |
| 通知用户被超出价 | 异步、解耦 | MQ、邮件/短信/站内信 |
| 成交后生成订单 | 分布式事务、最终一致 | 本地消息表、TCC、Saga |

**结论**:它能"自然地"逼出整个 Java 后端核心栈。你不是在为了用 Redis 而用 Redis,而是真的需要它。

## 三、八个阶段详细拆解

### 阶段 0:设计(本阶段)

**目标**:在不写一行代码的前提下,把"系统应该长什么样"想清楚。

**产出物**:`docs/` 目录下 10 份文档。

**学到的能力**:
- 需求分析方法:角色梳理 → 用例 → 功能列表 → 非功能指标。
- 架构思维:分层、模块、数据流、依赖方向。
- 技术选型方法论:为每一项技术回答"为什么是它,而不是别的"。

**关键产出**:
- 一份你能"看着讲 30 分钟"的架构图。
- 一份完整的数据库 ER 图与建表 SQL。
- 一份"全部接口列表"的 API 文档草稿。

> 论文写作的素材 80% 在这一步就要准备好。

### 阶段 1:工程骨架

**目标**:搭好后端多模块 Maven 工程 + 前端 Vue 工程,跑出"Hello World"。

**核心知识点**:
- Maven 多模块结构、`pom.xml` 继承与聚合、依赖管理(`dependencyManagement`)。
- Spring Boot 3 的自动装配、`@SpringBootApplication`、profile 多环境。
- Vite 工程结构、ESLint、Prettier、约定式路由。
- `application.yml` 配置中心化、敏感信息抽离。
- `.editorconfig`、`.gitignore`、`.gitattributes` 规范。

**学到的能力**:工程规范化,这是企业项目的"门面"。

---

### 阶段 2:用户与权限(认证授权)

**目标**:实现注册、登录、退出、修改密码、用户增删改查、基于角色的权限控制。

**核心知识点**:
- **Spring Security**:Filter 链、`UserDetailsService`、`PasswordEncoder`、自定义认证。
- **JWT**:无状态认证、Token 续签、黑名单(Redis 存储登出 token)。
- **RBAC**:用户—角色—权限三表,`@PreAuthorize` 注解式鉴权。
- **AOP 操作日志**:自定义注解 `@Log`,通过环绕通知记录"谁、什么时候、做了什么"。
- **统一异常**:`@RestControllerAdvice` + `@ExceptionHandler` + 自定义业务异常。
- **统一返回**:`Result<T>` 泛型封装。
- **参数校验**:`@Valid` + `BindingResult`,JSR-303 注解。
- **密码安全**:BCrypt 加盐哈希;不可逆。
- **登录限流**:Redis 计数,5 次失败锁 15 分钟。

**学到的能力**:Web 安全的基本功,任何后端都要做这一套。

---

### 阶段 3:商品与分类

**目标**:实现商品 CRUD、分类树、图片上传、富文本描述。

**核心知识点**:
- **MyBatis-Plus**:CRUD 抽象、条件构造器、分页插件、逻辑删除、自动填充(创建时间/创建人)。
- **分类树**:邻接表 vs 路径枚举 vs 闭包表的对比与选择。
- **图片上传**:本地 / MinIO / 阿里 OSS 的对比与切换;前端预签名 URL。
- **大字段(商品详情)**:为什么要拆表 / TEXT vs JSON 字段。
- **数据字典**:统一管理状态、类型枚举。

**学到的能力**:典型业务建模、文件存储、数据库优化基本功。

---

### 阶段 4:拍卖核心(高并发出价) ⭐

**目标**:实现"发布商品 → 用户出价 → 防止超出价 → 实时推送"。这是**整个项目的技术高潮**。

**核心知识点**:
- **业务规则**:出价必须 > 当前最高价 + 加价幅度;不能给自己出价;商品状态必须是"拍卖中"。
- **并发问题演示**:先用最朴素的"查询当前最高价 → 比较 → 写入"实现,用 JMeter 100 并发压测,亲眼看到"超出价 BUG"。
- **方案 A:数据库悲观锁** `SELECT ... FOR UPDATE`(讲清楚为什么不优雅)。
- **方案 B:数据库乐观锁**(version 字段 + UPDATE WHERE version=?)。
- **方案 C:Redis 分布式锁**(Redisson 的看门狗机制)。
- **方案 D 终极:Redis + Lua 脚本原子更新**(性能最好,生产首选)。
- **WebSocket(STOMP)**:出价成功后向商品房间广播,所有围观用户实时看到。
- **数据落库**:Lua 改 Redis 当前最高价 → MQ 异步把出价记录写 MySQL,削峰。

**学到的能力**:这是面试时最常被问、也最能拉开差距的部分。

---

### 阶段 5:异步与最终一致(RabbitMQ)

**目标**:把同步耗时操作改成异步,用 MQ 解决三个真实问题。

**核心场景**:
1. **被超出价通知**:用户 A 是当前最高,B 出更高价,需要通知 A "你已被超价"。出价主流程不应该等通知发完。
2. **拍卖到期结算**:商品挂出 24h 后自动结算,生成订单,通知中标人。用 RabbitMQ **死信队列(TTL+DLX)** 实现延迟消息。
3. **订单超时未支付**:订单 30 分钟未付款,自动取消、释放保证金。同样的延迟消息模式。

**核心知识点**:
- RabbitMQ 的 4 种 Exchange(direct/topic/fanout/headers)。
- **消息可靠性**三连:生产者 confirm + 消费者 ack + 持久化。
- **幂等消费**:消息可能重复,通过唯一业务 ID + Redis SETNX 去重。
- **死信队列**:延迟消息的经典实现。
- **Kafka 对比**:为什么有些场景(日志、埋点)用 Kafka 更合适;吞吐 vs 灵活路由。

**学到的能力**:分布式系统的"减压阀",企业项目几乎必备。

---

### 阶段 6:搜索(Elasticsearch)

**目标**:商品搜索框支持中文分词、相关性排序、按价格/分类聚合筛选。

**核心知识点**:
- ES 的核心概念:索引、文档、分片、副本、倒排索引。
- IK 分词器:`ik_smart` vs `ik_max_word`,自定义词典。
- 查询 DSL:bool、must/should/filter、match、term、range。
- 聚合查询:terms、range、avg。
- **数据同步**:MySQL → ES 的三种方案对比(双写 / Canal 监听 binlog / MQ 异步)。
- 搜索结果**高亮**与**分页(deep paging 问题)**。

**学到的能力**:大数据量下的搜索方案,以及"主存储 vs 检索引擎"的架构思想。

---

### 阶段 7:业务闭环

**目标**:订单 / 支付(沙箱)/ 收货 / 评价 / 信用分 / 保证金。

**核心知识点**:
- **状态机**:订单状态(待支付 / 已支付 / 已发货 / 已完成 / 已取消 / 申诉中)的合法迁移,使用状态模式或 Spring StateMachine。
- **分布式事务**:扣保证金 + 改商品状态 + 写订单,用本地消息表实现最终一致。
- **支付集成**:支付宝沙箱(可选)或自实现"模拟支付"。
- **评价系统**:买家评卖家、卖家评买家;信用分计算。
- **数据统计**:每日 / 每周成交额、热门分类、用户活跃度,使用 ES 聚合或 MySQL 窗口函数。

**学到的能力**:业务复杂度建模、分布式事务的工程取舍。

---

### 阶段 8:上线与运维

**目标**:用 Docker 容器化整个系统,Nginx 反代,真正在一台 Linux 上跑起来。

**核心知识点**:
- Dockerfile 多阶段构建(jar 镜像最小化)。
- docker-compose 编排:MySQL、Redis、RabbitMQ、ES、Nginx、Backend、Frontend。
- Nginx:反向代理、Gzip、静态资源缓存、HTTPS(Let's Encrypt)。
- **JMeter 压测报告**:出价接口 QPS、TPS、99 线响应时间。
- **性能优化**:索引优化、慢 SQL、JVM 参数(G1/ZGC)、连接池(HikariCP)。
- 日志采集(可选):ELK 或 Loki + Grafana。

**学到的能力**:从"代码能跑"到"系统能用"的最后一公里。

## 四、文档与代码的对应关系

每写一个模块的代码,都对应文档里的某一节。**文档先行**是企业的标准做法,论文写起来也轻松。

```text
docs/01 需求分析     →  全局参考
docs/02 技术选型     →  pom.xml 依赖确定
docs/03 架构设计     →  模块划分、包结构
docs/04 数据库设计   →  init.sql、Entity 类
docs/05 接口设计     →  Controller 层
docs/06 关键技术方案 →  Service 层(核心算法)
docs/07 安全与性能   →  Filter、AOP、缓存配置
docs/08 部署与运维   →  deploy/ 目录
docs/09 开发计划     →  GitHub Issues / Project
```

## 五、这份学习计划的"心法"

1. **不要跳着学**。看似你想直接写出价代码,但没有 JWT 你不知道是谁出价、没有数据库设计你不知道存哪。
2. **每学一个新技术,都要回答三问**:解决什么问题?不用它会怎样?替代方案有哪些?
3. **每个阶段都要写"思考笔记"**:遇到的坑、当时怎么想的、最后怎么解决的。这些都是论文素材,也是面试谈资。
4. **重视压测**。没有压测过的"高并发设计"都是空谈。在阶段 4 之后,务必用 JMeter 压一压"加锁前 vs 加锁后"。
5. **不要害怕重构**。第一版能跑通就行,发现问题再优化。先 work,再 better,最后 fast。
