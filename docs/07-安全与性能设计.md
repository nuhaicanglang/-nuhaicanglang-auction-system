## 一、安全设计

### 1.1 注册/登录多因子与会话

#### 1.1.1 注册门槛(防机器注册)

| 阶段 | 措施 |
|------|------|
| 前置 | 必填图形验证码(Hutool LineCaptcha,`captcha:<id>` Redis 5 分钟过期) |
| 邮箱校验 | 先调 `/auth/email-code/send` 发 6 位随机码,Redis `email:code:REGISTER:<email>` 5 分钟过期 |
| 限频 | 同邮箱 60s/次、10 次/天;同 IP 30 次/小时 |
| 验证顺序 | 图形码 → 邮箱码 → 唯一性 → 插表;任一失败不走下一步 |
| 邮件发送 | `JavaMailSender` + Thymeleaf 模板;走 `notifyExecutor` 异步;接口不等 SMTP |

#### 1.1.2 登录门槛

| 关注点 | 措施 |
|-------|------|
| 密码存储 | BCrypt 加盐(cost=10),禁明文/MD5 |
| 登录暴力破解 | Redis 计数,5 次/15 分钟锁定;IP + 用户双维度 |
| Token 防盗用 | JWT 短有效期(2h)+ Refresh Token(7d) |
| 登出失效 | JWT jti 加 Redis 黑名单 |
| 会话续签 | Refresh 滑动过期;7 天不活跃强制重登 |
| 多端登录 | 支持多设备,管理员可踢下线 |
| HTTPS | 强制 HTTPS,Nginx HSTS |
| 验证码渐进 | 同账号/IP 连续失败 ≥ 3 次 → 后续必填图形验证码(`fail:<key>` Redis 计数) |
| 黑名单拦截 | 用户 `status=2` 拉黑 → 登录报 `账号已被拉黑`,现有 JWT 全部加 Redis 黑名单,WebSocket 主动断开 |

### 1.2 授权(RBAC)

- 接口鉴权:`@PreAuthorize("hasAuthority('xx:yy')")`
- 数据权限:自定义 `@DataScope` 自动加 `WHERE created_by=?`
- 防越权:按 ID 操作前校验归属
- 管理员不可出价:Service 层硬性判定

### 1.3 输入校验与攻击防护

| 攻击 | 防护 |
|-----|------|
| SQL 注入 | MyBatis `#{}` 参数化;白名单校验排序字段 |
| XSS | 富文本 jsoup 白名单清洗 |
| CSRF | JWT in Header,不依赖 Cookie |
| 文件上传 | 校验后缀+MIME+magic number;UUID 化文件名 |
| URL 重定向 | 域名白名单 |
| ID 枚举 | 雪花 ID(无序) |

### 1.4 数据保护

- 敏感字段加密:身份证号 AES + 数据库密钥分离
- 列表脱敏:手机号 `13****1234`、邮箱 `a***@b.com`、姓名 `张*三`
- 日志脱敏:密码、token、身份证不打日志
- 备份加密:mysqldump → openssl aes-256-cbc

### 1.5 风控规则

- 出价频率:同用户同商品 1秒 1次
- 信用分阈值门槛(详见 01-需求 F-CREDIT 4.10.2 分段):
  - 60~79:不能一口价,保证金加倍,单次出价上限 5 万
  - 30~59:禁出价、禁发布,返错码 40006
  - <30:自动禁用账号,需管理员介入
- 异常出价:1分钟内出价 > 20 次告警
- IP 黑名单:同一 IP 短时间多账号登录拦截
- 用户黑名单:`sys_user.status=2` → BidValidator 拦截,返错码 40009

### 1.6 审计与告警

- 操作日志:`@Log` 切面记录敏感操作
- 关键事件:登录失败超阈值、强制下架、撤销出价 → 告警
- 数据对账:每天 02:00 跑 Redis 当前价 vs MySQL bid 表一致性检查

### 1.7 金融级资金审计 ⭐(论文亮点)

> “钱”与“信用”是最敏感的两类状态。本项目采用**双写入 + 同事务 + 多重防护**保证完全可审计。

#### 1.7.1 双写入原则(头号铁律)

任何以下字段变动，都必须在同一事务内同时完成：

| 主表字段 | 流水表 |
|---------|-------|
| `sys_user_extension.balance` | `biz_wallet_transaction` |
| `sys_user_extension.frozen_amount` | `biz_wallet_transaction` |
| `sys_user_extension.credit_score` | `biz_credit_log` |

```java
@Transactional(rollbackFor = Exception.class)
public void adjustWallet(WalletAdjustCmd cmd) {
    // 1. 行锁读余额(SELECT ... FOR UPDATE + version 乐观锁)
    UserExtension ext = extensionMapper.selectByUserIdForUpdate(cmd.getUserId());
    BigDecimal balanceBefore = ext.getBalance();
    BigDecimal frozenBefore  = ext.getFrozenAmount();

    // 2. 计算变动后值(余额不能为负,冻结不能超余额)
    ext.applyAdjust(cmd);

    // 3. 更新主表
    extensionMapper.updateByVersion(ext);

    // 4. 同事务写流水(后期审计可重放出 balance/frozen 的每一步变化)
    walletTxMapper.insert(WalletTx.from(cmd, balanceBefore, ext.getBalance(),
                                        frozenBefore, ext.getFrozenAmount()));

    // 中间任何一步抛异常 → 整个事务回滚,主表与流水永不脱节
}
```

#### 1.7.2 多重防护

| 防护点 | 实现 |
|-------|------|
| 二次验证 | 管理员调账需输本人登录密码,错误返 50201 |
| 限额 | 单次 ≤ `admin.wallet.adjust.max`(默认 100 万) |
| 隔离 | 管理员互不能调自己账户(50203) |
| 幂等 | 请求头 `X-Idempotent-Key` → `biz_wallet_transaction.client_request_id` 唯一索引 |
| 审计 | 调账同时写 `sys_oper_log`(双重保险:资金流水 + 系统操作日志) |
| 对账 | 定时任务校验 `SUM(balance + frozen)` 与 `SUM(wallet_transaction 收入 - 支出)` 一致 |

#### 1.7.3 并发控制

- 同一用户账户不可同时被两个管理员修改 → `SELECT ... FOR UPDATE` 行锁
- `sys_user_extension.version` 乐观锁避免 ABA
- 出价冻结走 Redisson 分布式锁(细粒度 `wallet:lock:<userId>`),超时 200ms

#### 1.7.4 信用分变动同原则

- 事件驱动：订单完成/评价/违规 → 发 MQ → `CreditService` 消费
- 同事务：`UPDATE credit_score` + `INSERT biz_credit_log`
- 幂等：MQ 消息带 `event_id`，`biz_credit_log.related_id` 索引防重复
- 多事件聚合：每日加分上限 +5，超出部分不入流水

## 二、性能设计

### 2.1 性能目标

| 指标 | 目标 |
|-----|------|
| 商品列表 P99 | < 200ms |
| 商品详情 P99 | < 300ms |
| 出价接口 P99 | < 500ms |
| 出价单机 QPS | ≥ 500 |
| 系统可用性 | 99.5% |

### 2.2 数据库优化

#### 2.2.1 索引
按 `04-数据库设计.md` 建索引;遵守最左前缀;避免索引列函数运算。

#### 2.2.2 慢查询
- `slow_query_log = ON`,阈值 1s
- pt-query-digest 周期分析
- EXPLAIN + EXPLAIN ANALYZE 验证执行计划

#### 2.2.3 连接池(HikariCP)
- `maximum-pool-size`:20(机器配置 ↑ 时调到 50)
- `connection-timeout`:5000ms
- `max-lifetime`:30 分钟(< MySQL `wait_timeout`)

#### 2.2.4 SQL 规范
- 禁 `SELECT *`,只取需要字段
- 禁 N+1:批量用 `IN`
- 大事务拆小,事务内禁远程调用
- 分页深翻页用 `WHERE id > lastId LIMIT N`(seek)

### 2.3 缓存优化

#### 2.3.1 缓存层级
1. Caffeine 进程内缓存:分类树、字典(改动罕见)
2. Redis 分布式缓存:商品详情、当前价、热门榜
3. CDN:静态资源(图片、JS、CSS)

#### 2.3.2 缓存与 DB 一致性
按 `06-关键技术方案.md` Cache Aside + 延迟双删。

#### 2.3.3 防三大问题
- 击穿:互斥锁重建 / 永不过期 + 后台刷新
- 雪崩:TTL 加随机扰动 (±2 min)
- 穿透:布隆过滤器 + 空值缓存(短 TTL 60s)

### 2.4 应用层优化

#### 2.4.1 异步化
- `@Async` 用于通知发送、ES 同步、统计累加
- 自定义线程池(命名、有界队列、拒绝策略)

**按业务场景隔离 4 个线程池**——防止一个场景狂冲低阶其他场景。

| Bean 名 | 场景 | core/max/queue | 拒绝策略 |
|--------|------|--------------|---------|
| `notifyExecutor` | 邮件、站内信、WS 推送 | 4 / 8 / 500 | CallerRuns |
| `logExecutor` | 操作日志、登录日志写入 | 2 / 4 / 2000 | CallerRuns |
| `esSyncExecutor` | 商品/出价同步到 ES | 2 / 4 / 1000 | CallerRuns |
| `statsExecutor` | 热点/热门统计、导出任务、定时对账 | 2 / 4 / 200 | CallerRuns |

> 拒绝策略统一用 **`CallerRunsPolicy`**：队列满时降级到调用线程执行，不丢任务（现金路径高优先）。

```java
@Bean("notifyExecutor")
public ThreadPoolTaskExecutor notifyExecutor() {
    ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
    exec.setCorePoolSize(4);
    exec.setMaxPoolSize(8);
    exec.setQueueCapacity(500);
    exec.setThreadNamePrefix("notify-");
    exec.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    exec.initialize();
    return exec;
}
```

#### 2.4.2 减少 IO
- 合并多次 Redis:Pipeline / Lua
- 合并多次 DB:`saveBatch`、`IN` 查询
- 列表接口预加载关联数据(避免 N+1)

#### 2.4.3 序列化
- 接口 JSON:Jackson 默认即可,关闭 `FAIL_ON_UNKNOWN_PROPERTIES`
- Redis 缓存:Jackson `GenericJackson2JsonRedisSerializer`
- 大对象用 Kryo / Protobuf(本项目暂不引入)

### 2.7 RabbitMQ 调优

- `prefetch: 5~10`:消费者一次最多处理几条
- 集群模式开启镜像队列(本项目单机不必)
- 消息体保持小(≤ 4KB),大数据走 Redis 引用

### 2.8 ES 调优

- `refresh_interval: 30s`:写多读少时降低刷新频率
- 写入用 bulk
- 别用深翻页(`from + size > 10000`),要 deep paging 就 `search_after`
- mapping 中合理选 keyword vs text(精确 vs 分词)

### 2.9 前端性能

- 路由懒加载(动态 import)
- 图片懒加载 + WebP
- 商品列表虚拟滚动(超过 100 条时)
- Element Plus 按需引入
- Vite build:`splitChunks` 分包,vendor 长期缓存
- HTTP 请求合并、防抖(搜索框 300ms)

### 2.10 Nginx 优化

```nginx
gzip on;
gzip_types text/css application/javascript application/json;
gzip_min_length 1k;

# 静态资源强缓存
location ~* \.(js|css|png|jpg|jpeg|gif|webp|woff2)$ {
    expires 30d;
    add_header Cache-Control "public, immutable";
}

# 限流(每秒 10 个请求,突发 20)
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
location /api/ {
    limit_req zone=api burst=20 nodelay;
    proxy_pass http://backend;
}

# WebSocket 长连接
location /api/v1/ws {
    proxy_pass http://backend;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 3600s;
}
```

## 三、可观测性(监控)

### 3.1 应用指标

- Spring Boot Actuator + Micrometer
- 关键指标:
  - `http.server.requests`:接口耗时分位数
  - `bid.success.qps`:出价成功 QPS
  - `bid.fail.qps`:出价失败 QPS
  - `mq.consume.lag`:堆积消息数
  - `cache.hit.ratio`:缓存命中率
  - `executor.<name>.active` / `.queueSize` / `.completed`：4 个线程池状态 (Actuator `/actuator/metrics/executor.*`)
  - `wallet.adjust.qps` / `credit.change.qps`：资金变动频率
  - `wallet.balance.total`：全平台总余额（定时采集，用于对账及异常告警）

### 3.2 日志规范

```xml
<!-- logback-spring.xml -->
<pattern>
  %d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level [%X{traceId}] %logger{50} - %msg%n
</pattern>
```

- INFO:业务关键节点(下单、出价成功、登录)
- WARN:可恢复异常(参数非法、资源不存在)
- ERROR:系统异常,必须打全栈
- 敏感数据(密码、token、身份证)绝不打

### 3.3 链路追踪

- 请求入口生成 `traceId`(UUID 短码)写入 MDC
- Header `X-Trace-Id` 透传(单体内无跨服务,主要用于日志关联)

### 3.4 告警(可选)

- Spring Boot Admin 监控面板
- Prometheus + AlertManager
- 关键告警:
  - 出价接口 5xx > 5/分钟
  - MQ 堆积 > 1000
  - DB 连接池使用率 > 90%
  - JVM 老年代 > 80%

## 四、压测方案

### 4.1 工具:JMeter

### 4.2 场景设计

| 场景 | 配置 | 验证 |
|-----|------|------|

### 4.3 压测前准备

```bash
# 关闭日志 INFO,或异步日志(AsyncAppender)
# 数据预热:跑一遍场景,让 JVM JIT 热起来
```

### 4.4 压测报告必备

- TPS / QPS 曲线
- 响应时间分位数(P50/P90/P99)
- 错误率
- 服务器 CPU / 内存 / 磁盘 IO / 网络
- DB / Redis / MQ 资源使用
- JVM GC 次数与停顿

### 4.5 性能瓶颈定位思路

1. 接口慢 → APM 看每段耗时(controller / service / SQL / Redis / MQ)
2. CPU 高 → Arthas `profiler start`,火焰图
3. 内存高 → 堆转储 + MAT
4. SQL 慢 → 慢日志 + EXPLAIN
5. 锁竞争 → `jstack`(线程阻塞栈)、Arthas `thread -b`

## 五、容灾与降级

### 5.1 关键依赖故障策略

| 依赖 | 故障表现 | 降级 |
|-----|---------|------|
| Redis 不可用 | 缓存全部 miss | 限流 + 降级到 DB(短时,告警) |
| MQ 不可用 | 异步流程阻塞 | 写本地消息表,定时重投 |
| ES 不可用 | 搜索不可用 | 降级到 MySQL `LIKE`(性能差但能用) |
| DB 主库故障 | 写不可用 | 拒绝写请求,只读;接入只读副本 |

### 5.2 熔断(可选)

引入 Resilience4j:
```java
@CircuitBreaker(name = "esSearch", fallbackMethod = "fallbackSearch")
public PageResult<ItemVO> search(SearchQuery q) { ... }
public PageResult<ItemVO> fallbackSearch(SearchQuery q, Throwable e) {
    return mysqlFallbackSearch(q);
}
```

### 5.3 数据兜底

- 出价记录:Redis 队列 + 定时任务对账
- 订单状态:状态机 + 定时扫描"卡住"的订单
- 保证金账本:每日对账(冻结 + 余额 = 充值 - 消费)
