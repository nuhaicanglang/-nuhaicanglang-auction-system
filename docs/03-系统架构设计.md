## 二、模块划分(后端多模块 Maven)

### 2.1 模块清单

```text
auction-backend (parent pom,只做依赖管理,不出 jar)
├── auction-common         # 工具、统一返回、错误码、常量、注解定义
├── auction-framework      # 基础设施配置(Security/Redis/MQ/ES/MyBatis/Web)
├── auction-system         # 系统模块:用户、角色、权限、字典、操作日志
├── auction-business       # 业务模块:分类、商品、拍卖、出价、订单、保证金、评价、收藏
├── auction-search         # 搜索模块:ES 索引管理、检索接口
├── auction-job            # 定时任务模块
├── auction-mq             # 消息生产/消费(队列声明、Listener)
└── auction-admin          # 启动入口,聚合所有模块,打成可执行 jar
```

### 2.2 依赖方向(必须严格遵守)

```text
auction-admin
   ├── auction-system
   ├── auction-business
   ├── auction-search
   ├── auction-job
   └── auction-mq

auction-business / auction-system / auction-search / auction-job / auction-mq
   └── auction-framework

auction-framework
   └── auction-common

auction-common  (无依赖)
```

**禁止**:`auction-system` 依赖 `auction-business` 或反向(横向不依赖)。如果两个模块需要共享逻辑,**下沉到 framework 或 common**。

### 2.3 各模块职责详解

| 模块 | 包路径 | 主要内容 |
|------|--------|---------|

### 2.5 命名约定

- **接口/实现**:`XxxService` / `XxxServiceImpl`。
- **Entity 不直接出接口**:对外用 DTO/VO,避免数据库字段泄露。
- **Request 后缀**用于入参,**VO 后缀**用于出参,**DTO**用于服务间内部传递。
- **Converter** 用 MapStruct,放对应模块的 `converter` 包。

### 3.2 场景二:出价(高并发核心)⭐

```text
Browser  Nginx  SpringBoot   Redis(Lua)     RabbitMQ      MySQL    WebSocket
  │       │        │             │              │            │          │
  │── POST /bid ───▶             │              │            │          │
  │       │        │── 1. JWT 鉴权              │            │          │
  │       │        │── 2. 限流(Redis 令牌桶)──▶│            │          │
  │       │        │  ◀─── 通过 ───            │            │          │
  │       │        │── 3. 校验保证金/状态(MySQL)──────────▶│          │
  │       │        │  ◀──── 校验通过 ──────────────────────│          │
  │       │        │── 4. EVALSHA Lua 脚本 ──▶│            │          │
  │       │        │      (比较 + 更新最高价 + 入队)        │          │
  │       │        │  ◀── ret=1(成功)  ───   │            │          │
  │       │        │── 5. 发出价记录消息 ─────────────▶    │          │
  │       │        │── 6. 发"被超价通知"消息 ────────────▶  │          │
  │       │        │── 7. WS 广播新最高价 ──────────────────────────▶│
  │       │        │                                                   │── 推送给所有围观者
  │◀── 200 ───────│                                                   │
                                            │            │
                       MQ 异步消费 ───────  │            │
                       (1) 出价记录持久化 ──────────────▶│
                       (2) 站内信/邮件 ────              │
                       (3) ES 同步 ─────                  │
```

**重点**:同步链路只做"鉴权 → 限流 → 校验 → Lua 出价 → 推送",**3 次网络往返内返回**。所有持久化和通知是异步。

### 3.3 场景三:拍卖到期自动结算(延迟消息)

```text
发布商品时:
  Service ── 生产消息(payload=itemId, TTL=拍卖时长) ──▶ 延迟交换机
                                                          │
                                                          ▼
                                                    ttl-queue(无消费者)
                                                          │ (TTL 到期)
                                                          ▼
                                                  死信交换机 DLX
                                                          │
                                                          ▼
                                                  settle-queue(有消费者)
                                                          │
                                                          ▼
拍卖结算消费者:
  1. 加 Redisson 锁(key=item:settle:<itemId>),防止重复结算
  2. 查询商品状态,若已结束(可能被一口价提前)直接 ack
  3. MySQL 事务:
     - SELECT 商品 FOR UPDATE
     - 状态置为 已结束/已成交/流拍
     - 记录最终价、中标人
     - 释放其他出价者的保证金(insert deposit_release)
  4. 若有中标人:生成订单(status=待支付),投递订单超时延迟消息(TTL=30min)
  5. 发"中标通知/流拍通知"消息(给具体用户)
  6. 发"广播商品状态变更"WebSocket 消息
  7. 发"ES 同步状态"消息
```

**重点**:每一步都可能失败,因此整个消费者必须**幂等**(重入锁 + 状态判断),失败时 nack 重入队列。

## 五、横切关注点(贯穿所有模块)

### 5.0 请求生命周期与组件分工 ⭐

> 一个 HTTP 请求从浏览器到达 Controller,会**依次穿过 Nginx → Filter → Interceptor → AOP → Controller** 四层。每层职责不同,**不可互相替代**。

#### 5.0.1 完整链路

```text
Browser
  │ HTTPS
  ▼
┌─────────────────────────────────────────────────┐
│ Nginx(进程外网关角色)                            │
│  • HTTPS 卸载 / HSTS                              │
│  • gzip / br 压缩                                 │
│  • 静态资源 (/dist)                               │
│  • 反向代理 /api → backend:8080                   │
│  • IP 粒度限流 limit_req_zone                     │
│  • 上传大小限制 client_max_body_size              │
└─────────────────────────────────────────────────┘
  │ HTTP(内网)
  ▼
Tomcat Connector
  │
  ▼
┌─ Servlet Filter Chain ─────────────────────────┐
│ 1. CorsFilter                                   │ 跨域
│ 2. TraceIdFilter         (生成 X-Trace-Id)     │ MDC 注入
│ 3. RepeatableReadFilter  (请求体可重读)         │ 给 AOP 读 body 用
│ 4. JwtAuthenticationFilter                      │ 解 JWT + 查 Redis 黑名单
│ 5. Spring Security FilterChain (内置多层)       │ 认证 + 权限决策
└─────────────────────────────────────────────────┘
  │
  ▼
DispatcherServlet
  │
  ▼
┌─ Spring HandlerInterceptor ───────────────────┐
│  PermissionInterceptor (可选,主要让 @PreAuthorize)│
└─────────────────────────────────────────────────┘
  │
  ▼
┌─ Spring AOP @Around ─────────────────────────┐
│  @RateLimit  (Lua 限流,按 userId/itemId)      │
│  @Idempotent (X-Idempotent-Key 防重)          │
│  @Log        (操作日志,异步走 logExecutor)    │
│  @Transactional (事务)                        │
└─────────────────────────────────────────────────┘
  │
  ▼
Controller → Service → Mapper → MySQL/Redis/MQ/ES
```

#### 5.0.2 四层选择口诀

```text
HTTP 层、无关业务  → Filter      (CORS、Trace、JWT 解析)
要 HandlerMethod   → Interceptor (粗粒度权限拦截)
要方法参数和返回值 → AOP         (日志、限流、幂等、事务)
跨进程、跨服务的活 → Nginx       (HTTPS、静态、反代、IP 限流)
```

#### 5.0.3 为什么不用 API 网关(Spring Cloud Gateway)

| 场景 | 是否需要网关 | 本项目方案 |
|------|----------|----------|
| 多微服务统一入口 | 需要 | ❌ 本项目是单体多模块,只有 1 个后端进程 |
| 路由/反代 | 需要 | ✅ Nginx 反代 `/api` |
| HTTPS 卸载 | 需要 | ✅ Nginx |
| 限流熔断 | 需要 | ✅ Nginx 粗限流 + AOP `@RateLimit` 细限流 |

**结论:Nginx 已经承担了网关 80% 的职责,且零额外进程开销。** 等到 6.5.7 微服务化第 3 步,才上 Spring Cloud Gateway。

#### 5.0.4 Nginx 与 Filter 的"分工边界"(易混淆点)

| 任务 | Nginx | Filter | 为什么 |
|------|-------|--------|-------|
| HTTPS、gzip、静态、IP 限流 | ✅ | — | C 语言、零拷贝、内核态最快 |
| **JWT 解析 + Redis 黑名单查询** | ❌ | ✅ | 需查 Redis,Nginx 要装 OpenResty + Lua,得不偿失 |
| **TraceId 写入 MDC** | ❌ | ✅ | MDC 是 Logback 的 ThreadLocal,Nginx 写不进 |
| **按 userId 限流**(细粒度) | ❌ | ✅(走 AOP) | 要先解 JWT 才知道 userId |
| **业务错误返回**(`{"code":40001}`) | ❌ | ✅ | Nginx 只能返 HTTP 状态码,前端要做两套解析 |
| 跨域 CORS | ✅或✅ | ✅或✅ | 二选一,本项目交给 Spring CorsFilter,前端调试时灵活 |

> 一句话:**Nginx 干"粗活"(网络层、协议层),Filter 干"细活"(看 Java 对象、查 Redis)**。

### 5.1 全局异常处理

```text
@RestControllerAdvice
GlobalExceptionHandler
├── handleBizException(BizException)            → 业务错误码透传
├── handleValidationException(MethodArgumentNotValidException)  → 参数校验错误
├── handleAccessDenied(AccessDeniedException)   → 403
├── handleAuthentication(AuthenticationException) → 401
├── handleSqlException(SQLException)            → 屏蔽错误细节,统一 500
└── handleAll(Throwable)                         → 兜底,日志要打全栈
```

返回固定结构:
```json
{ "code": 40001, "msg": "金额不足", "data": null, "traceId": "abc" }
```

### 5.2 统一返回封装

```text
Result<T>
  static <T> Result<T> ok(T data)
  static <T> Result<T> ok()
  static Result<Void> fail(ErrorCode code)
  static Result<Void> fail(int code, String msg)

PageResult<T> extends Result<List<T>>
  total, pageNum, pageSize
```

### 5.3 链路追踪(traceId)

- `Filter` 拦截每个请求,从 Header 取 `X-Trace-Id`,没有就生成。
- 写入 `MDC`,日志格式中输出 `%X{traceId}`。
- 跨服务时从 Header 透传(本项目单体,主要用于日志关联)。

### 5.4 操作日志切面

```text
@Around("@annotation(com.auction.common.annotation.Log)")
public Object aroundLog(ProceedingJoinPoint pjp, Log log) {
    1. 记录入参、当前用户、IP、UA
    2. 执行业务
    3. 记录返回值/异常、耗时
    4. 异步入库 sys_oper_log
}
```

### 5.5 审计字段自动填充

MyBatis-Plus `MetaObjectHandler`:
- `created_at`、`created_by` 在 INSERT 时自动填(从 SecurityContext 取当前用户 ID)。
- `updated_at`、`updated_by` 在 UPDATE 时自动填。

## 六、扩展性预留

### 6.1 接口版本

URL 前缀 `/api/v1/...`,新版本接口走 `/api/v2/...`,老版本可平滑下线。

### 6.2 数据库分库分表预留

- 出价表 `auction_bid` 在 DDL 中加 `tenant_id` 占位字段(本项目恒为 0),将来按 `user_id % N` 分表时不用改代码结构。
- 订单表 `business_order` 同上。

### 6.3 微服务拆分预留

- 模块边界已经按"用户域"和"业务域"清晰划分,将来可:
  - `auction-system` → user-service。
  - `auction-business` → 拆 item-service / bid-service / order-service。
  - 模块间调用现在是方法调用,届时改为 Feign/gRPC。

### 6.4 多租户预留

- 所有 Entity 预留 `tenant_id`,默认 0;将来支持 SaaS 时按 tenant 隔离。

### 6.5 可扩展性评估(论文专章素材)⭐

> 答辩时"可扩展性"是必问题。本节系统化回答"未来要变,改哪里、改多少"。

#### 6.5.1 横向扩展(单实例 → 多实例)

| 维度 | 当前实现 | 多实例改造点 | 工作量 |
|------|---------|------------|------|
| **会话** | JWT 无状态 | ✅ 无需改 | - |
| **WebSocket** | `SimpleBroker`(进程内) | 切换 `enableStompBrokerRelay` → RabbitMQ STOMP plugin,所有实例的客户端订阅通过 RabbitMQ 中继 | 0.5 天 |
| **缓存** | Redis 集中式 | ✅ 已是分布式 | - |
| **分布式锁** | Redisson + Lua | ✅ 多实例天然安全 | - |
| **定时任务** | `@Scheduled` 每实例都跑会重复 | 引入 ShedLock(用 Redis/MySQL 选主)或换 XXL-Job | 0.5 天 |
| **文件上传** | MinIO | ✅ 共享存储 | - |
| **Session 同步** | 无 | ✅ 无需 | - |

**结论**:**90% 工作已做完**,只需改 WebSocket 与定时任务,1 天内可水平扩展。

#### 6.5.2 配置中心化

- 当前:`sys_config` 表 + Redis 缓存 + `/admin/configs/{key}` 热更新 → 实现了"配置中心的 80% 价值"。
- 升级到 Nacos/Apollo:把 `sys_config` 表的读取换成 Nacos client,**Service 层零改动**(预留 `IConfigService` 抽象)。

#### 6.5.3 灰度发布

- Nginx `map` 指令按 cookie/header 分流到不同后端 upstream。
- URL 已经是 `/api/v1/...`,新版本走 `/api/v2/...`,Nginx 按比例转发。
- DTO 字段**只增不删**保证向后兼容。

#### 6.5.4 监控可观测扩展

- 当前:Spring Boot Actuator + Micrometer → Prometheus(可选)。
- 升级 SkyWalking/OpenTelemetry:Agent 探针接入,**业务代码零侵入**。
- 日志统一格式(JSON + traceId)→ Filebeat 采集到 Elasticsearch → Kibana 可视化(ELK 栈)。

#### 6.5.5 业务扩展点(已用扩展模式预埋)

| 想加 | 怎么加 | 影响范围 |
|------|------|---------|
| 新拍卖类型(荷兰式/密封式) | 加 `AuctionStrategy` 子类,注册为 Bean | **0 处主流程修改** |
| 新支付方式 | 加 `PaymentAdapter` 子类(适配器模式) | 0 处主流程修改 |

#### 6.5.6 数据规模扩展

| 数据 | 当前容量 | 扩容路径 |
|------|---------|---------|
| 出价记录 | 单表 | 按 `item_id % N` 分表(`tenant_id` 已预留),单表 1 千万行内不分 |
| 订单 | 单表 | 同上 |
| 商品 | 单表(<10 万) | 暂不需要;接近时按 `created_at` 分区 |

| 站内信 | 单表 | 按 `user_id` 分表 |

#### 6.5.7 微服务拆分演进路线

```text
当前(单体多模块)
   │
   ├─► 第 1 步:auction-system 独立 → user-service(JWT 仍共享密钥)
   │
   ├─► 第 2 步:auction-business 拆 → item-service / bid-service / order-service
   │       Service 间调用改 Feign/gRPC,Bean 改成接口 + 远程实现
   │
   └─► 第 3 步:网关化 → Spring Cloud Gateway 统一入口
                       Nacos 服务注册发现
                       Sentinel 统一限流熔断
```

**关键**:**模块间不直接 new 对方 Bean**,只通过 Spring `@Autowired` 接口,这样未来替换为 Feign 客户端时,**业务代码零修改**。
